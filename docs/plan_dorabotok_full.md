**Дата:** 2026-07-24
**Основание:** `docs/audit_deep_national_platform_2026-07-24.md` (глубокий доказательный аудит).
**Контекст:** MVP = пилот = ближайшая цель запуска (несколько школ/знакомых, под контролем заказчика). Национальный режим (школы/Минпросвещения) — отдельная поздняя фаза.

> **Легенда приоритета:** 🔴 блокер запуска пилота · 🟠 важно к запуску · 🟡 вскоре после · ⚪ средний/долгий срок.

## ФАЗА A — ДО ЗАПУСКА ПИЛОТА (безопасность, доверие, психобезопасность)

Это то, без чего нельзя пускать реальных детей. Идём строго по порядку — каждый следующий шаг переиспользует предыдущий.

### A1. 🔴 Единый контроль доступа к данным (IDOR / OWASP A01:2025)
- **Цель:** чужой не может открыть чужой отчёт/результат.
- **Находка:** S-1, S-2, S-3.
- **Где:** `app/api/v1/diagnostic/results/route.ts`, `.../register-session`, `.../next-question`, `.../submit-answer`, `.../delete-last-answer`, `app/api/v1/coach/chat/route.ts`.
- **Шаги:**
  1. Создать серверный guard `app/lib/authz/requireOwnedResource.ts`: определить principal (`auth.api.getSession` **на сервере**, не клиент) → выбрать ресурс из БД **с `where userId = principal`** → только потом читать/писать Redis.
  2. Привязать ключи кэша к владельцу: `report:<userId>` / `session:<userId>` вместо произвольного `session_id`, чтобы cache-hit не обходил проверку.
  3. Убрать `user_id` из публичного API `next-question`; `register-session` перестаёт выводить личность из `coachSessionId`.
  4. Несоответствие владельца → `404` (не `403`).
- **Готово когда:** негативные E2E зелёные: «Алиса не видит отчёт Боба» (cache hit / финальный / промежуточный), «чужой `session_id` на диагностике → отказ», позитивный happy-path не сломан.
- **Зависит от:** —. → детали в `docs/implementation_plan_p0.md`.

### A2. 🔴 Безопасное слияние гостевой сессии
- **Цель:** нельзя присвоить чужую гостевую сессию, подставив её ID.
- **Находка:** S-3.
- **Где:** `app/api/v1/coach/chat/route.ts` (merge-блок ~123-190), создание гостевой сессии.
- **Шаги:**
  1. При старте гостя ставить одноразовую подписанную `HttpOnly; Secure; SameSite=Lax` куку-capability на `sessionId`.
  2. Merge разрешать только при валидной capability на этот `sessionId`; после merge — гасить (one-time).
  3. Обернуть merge (перепривязка + удаление старых + `mergedInto`) в один `prisma.$transaction`.
- **Готово когда:** E2E cross-user (нельзя присвоить чужую сессию) и growth-race (параллельный merge не теряет/не дублирует) зелёные; свой гость сливается, имя переносится.
- **Зависит от:** A1 (общий principal).

### A3. 🔴 Кризисный протокол (Safety, ISO/IEC 25010:2023)
- **Цель:** если подросток в диалоге пишет о суициде/насилии/остром стрессе — платформа безопасно останавливается и направляет к помощи.
- **Находка:** P-1.
- **Где:** `app/lib/coach/prompts.ts`, `app/api/v1/coach/chat/route.ts`, новый `app/lib/coach/safety.ts`.
- **Шаги:**
  1. С подростковым психологом согласовать: список сигналов distress, безопасный текст ответа, проверенные контакты (детский телефон доверия 8-800-2000-122 и т.п. — подтвердить актуальность).
  2. `safety.ts`: детектор сигналов (правила + опционально проверка моделью) на каждое сообщение пользователя.
  3. При срабатывании: коуч **прерывает** профориентационный сценарий, не играет терапевта, проявляет заботу, предлагает доверенного взрослого, показывает контакты.
  4. Явно НЕ передавать этот эпизод как «профориентационный факт» в профиль/отчёт.
- **Готово когда:** тест-кейсы с distress-фразами → безопасный ответ + контакты; обычный поток не задет ложными срабатываниями (проверить на нейтральных фразах).
- **Зависит от:** — (можно параллельно A1/A2). **Требует ревью психолога.**

### A4. 🟠 Telegram webhook fail-closed + токен не в URL
- **Цель:** вход из мессенджера нельзя подделать; session token не утекает.
- **Находка:** S-4, S-5.
- **Где:** `app/api/webhooks/telegram/route.ts`, `app/api/auth/telegram/route.ts`, `.../telegram/callback/route.ts`.
- **Шаги:**
  1. Webhook возвращает `503`, если `TELEGRAM_WEBHOOK_SECRET` не задан; `X-Telegram-Bot-Api-Secret-Token` сверять всегда.
  2. Ссылку входа строить только с одноразовым `exchange_token` (модель `AuthExchangeToken` уже есть); убрать `?token=<sessionToken>` из URL и логов.
- **Готово когда:** webhook без валидного секрета → отказ; replay → отказ; в ссылке только exchange_token; вход из Telegram работает end-to-end.
- **Зависит от:** —.

### A5. 🟠 Privacy-страница по факту + лёгкое согласие родителя
- **Цель:** страница не врёт про данные; для пилота собрано согласие родителя.
- **Находка:** L-2 (privacy), L-1 (пилотный вариант).
- **Где:** `app/privacy/page.tsx`, форма регистрации.
- **Шаги:**
  1. Переписать privacy под реальную схему: имя, класс, **email, телефон, Telegram/MAX ID, IP/user-agent, переписка с коучем, ответы тестов, отчёт, связь родитель-ребёнок**; цель, срок хранения, как удалить.
  2. Добавить при регистрации галочку «я родитель/законный представитель или у меня есть его согласие» + короткий понятный текст. Для пилота этого достаточно (полный флоу — фаза D).
- **Готово когда:** privacy соответствует БД; регистрация не проходит без отметки о согласии.
- **Зависит от:** —. **Текст согласия желательно показать юристу перед пилотом.**

### A6. 🟡 Инцидент-гигиена перед онбордингом
- **Цель:** чистый старт.
- **Где:** Redis, БД.
- **Шаги:** после A1–A2 очистить старые небезопасные ключи `report:*`/`session:*`; проверить логи на утёкшие в URL токены; при находках — ротация секретов.
- **Готово когда:** старых ключей нет, логи чисты.
- **Зависит от:** A1, A2, A4.

## ФАЗА B — ЯДРО ПРОДУКТА: СВЯЗНОСТЬ ДАННЫХ И ЛОГИКА ПОДАЧИ

Сразу после безопасности — то, что делает продукт цельным, а не набором тестов. Это твой ключевой запрос про «всё стекается в одно».

### B1. 🟠 D-1: Сквозная связность данных (единый источник истины)
- **Цель:** любой ответ любой сессии/теста дополняет общий профиль; всё стекается в Колесо, Пирамиду, матчинг и отчёт из одного места.
- **Находка:** C-4, M-2.
- **Где:** `app/lib/profile/layers.ts`, `buildSummaryProfile`, `app/lib/profile/consistency.ts`, `DigitalProfile.summary`, поверхности отчёта (`app/components/report/*`), `professionMatch.ts`.
- **Шаги:**
  1. **Инвентаризация:** пройти по `layers.ts`, у каждого из 40 полей `// TODO: нет источника` решить — назначить источник (вопрос коуча / пункт теста) или осознанно удалить дубль. Итог: у **каждого** поля профиля есть вход.
  2. **Единый конвейер:** ответ → `extractedData`/`diagnosticAnswers` → `buildSummaryProfile` → `DigitalProfile.summary`. Все четыре поверхности (Колесо талантов, Пирамида Дилтса, тесты-для-матчинга, финальный отчёт) читают **из `summary`**, не перевычисляя самостоятельно.
  3. **Инкрементальный Consistency Index:** `consistency.ts` пересобирается при каждом новом ответе; коуч и тест дополняют друг друга, противоречия подсвечиваются как probe (вопрос на уточнение), а не теряются.
  4. **Тест-инвариант:** «нет поля профиля без источника» + «каждый введённый ответ отражается хотя бы в одной из четырёх поверхностей». Регрессионный тест на не-разрозненность.
- **Готово когда:** инварианты зелёные; демо: ответ в коуче меняет и Колесо, и Пирамиду, и матчинг, и отчёт.
- **Зависит от:** A1 (данные читаются владельцем безопасно).

### B1a. 🟠 Колесо и Пирамида в личном кабинете (к цифровому профилю)
- **Цель:** Колесо талантов и Пирамида Дилтса доступны в **личном кабинете**, привязаны к цифровому профилю, а не только в отчёте.
- **Находка:** решение заказчика 2026-07-24; связано с B1 (единый источник).
- **Где:** `app/components/report/RadarChart.tsx`, `DeepSessionCard.tsx` (Пирамида), личный кабинет, `DigitalProfile.summary`.
- **Шаги:** переиспользовать компоненты Колеса/Пирамиды в кабинете; данные тянуть из `DigitalProfile.summary` (тот же источник, что отчёт).
- **Готово когда:** в кабинете видны Колесо и Пирамида, обновляются вместе с профилем.
- **Зависит от:** B1.

### B1b. 🟠 Трек H — достройка паспортов профессий (бо́льшая часть уже есть)
- **Уже реализовано (проверено в коде 2026-07-24):** `salary` 204/204, `fact` 204/204 (`professionExtras.ts`), `subjects` в `professions_db.ts`, блок «Почему тебе подходит» + разбивка по осям и `fact` в карточке (`RpgProfessionCard.tsx`).
- **Осталось для MVP:**
  1. `educationPath` (куда поступать) — сейчас **0/204**, наполнить курировано;
  2. `outlook` (перспективы/тренд востребованности) — сейчас **0/204**, наполнить с датировкой источника;
  3. честные оценки вместо псевдо-процента 58–99% (делается в **B2**).
- **Находка:** C-2, C-1; этап 8 docs/26.
- **Где:** `app/data/professionExtras.ts` / `professions_db.ts` (`educationPath?`, `outlook?` — поля уже в интерфейсе), `RpgProfessionCard.tsx` (рендер уже готов, показывает при наличии данных).
- **Готово когда:** `educationPath` и `outlook` заполнены для всех профессий; в карточке видны «Путь» и «Перспективы».
- **Зависит от:** B2 (честная подача оценок).

### B2. 🟠 Честная подача результата (без ложных процентов)
- **Цель:** подросток видит «направления для исследования», а не псевдо-вероятность «92% подходит».
- **Находка:** C-1, C-3, P-2 (анти-детерминизм).
- **Где:** `app/lib/profile/professionMatch.ts:283-285`, компоненты отчёта, `app/lib/coach/prompts.ts`.
- **Шаги:**
  1. Заменить визуализацию «X%» на: направление + **по каким осям совпало** (breakdown уже считается) + что пока неизвестно + следующая профпроба.
  2. Ранжирование и архетипы **сохранить** — убираем только вводящий в заблуждение процент.
  3. В промпт выдачи добавить явный анти-детерминизм: «это твои сильные направления и вектор, а не приговор; можно передумать и перепройти».
- **Готово когда:** в отчёте нет «% пригодности» как вероятности; формулировки — про направления и объяснимость.
- **Зависит от:** B1 (единый профиль).

### B3. 🟡 Границы «профориентация ≠ терапия»
- **Цель:** платформа не выдаёт себя за психотерапию; личная переписка не уходит родителю по умолчанию.
- **Находка:** P-3, P-4.
- **Где:** отчёт, кабинет родителя (когда появится), `prompts.ts`.
- **Шаги:** дисклеймер «это инструмент самопознания, не замена специалиста»; зафиксировать, что родителю по умолчанию идёт **сводка**, а не сырой transcript.
- **Готово когда:** дисклеймер на месте; поведение передачи данных родителю специфицировано.
- **Зависит от:** B1.

## ФАЗА C — ДИЗАЙН, ДОСТУПНОСТЬ, КОНТЕНТ

### C1. 🟡 Ручная доступность с ассистивными технологиями (WCAG 2.2 AA)
- **Цель:** реальная доступность, не только автопроверка.
- **Находка:** U-1, U-2, U-3.
- **Где:** все потоки (коуч, тест, отчёт, 3D-аватар), `next.config.mjs`, компоненты.
- **Шаги:**
  1. Ручной проход NVDA + VoiceOver по потокам регистрация → коуч → тест → отчёт.
  2. Контраст ≥ 4.5:1 (крупный ≥ 3:1) для текста, в т.ч. на 3D-аватаре и «стеклянных» плашках.
  3. Новые критерии 2.2: **Target Size** (интерактивные цели достаточного размера), **Accessible Authentication** (вход через QR/Telegram без когнитивного теста/запоминания), Focus Not Obscured.

- **Готово когда:** ручной чек-лист AA закрыт; axe в CI без критики.

## ФАЗА E — МЕТОДОЛОГИЯ И ДОКАЗАТЕЛЬНОСТЬ (долгосрок)

### E1. ⚪ Реестр методик
- **Цель:** у каждого инструмента (RIASEC, BFI, ICAR, PVQ, VIA…) — версия, источник, лицензия, перевод, возраст, ключ, допустимая интерпретация.
- **Находка:** M-3, M-1.
- **Готово когда:** реестр заполнен; расхождения перевода/лицензии закрыты.

### E2. ⚪ Психометрический пилот
- **Цель:** заменить эвристику «reliability» реальными метриками.
- **Находка:** M-1.
- **Шаги:** content review; Cronbach α/ω; факторный анализ; DIF; test-retest; нормирование на выборке РФ 13–18. (Порог α≥0.70 — минимум; для решений ближе к 0.80+.)
- **Готово когда:** ключевые шкалы валидированы и нормированы; интерпретации опираются на нормы.

### E3. ⚪ Модернизация моделей (не урезание)
- **Находка:** M-4, M-5.
- **Шаги:** 3-буквенный код Холланда + Congruence/Differentiation Index; при желании — шкала CAAS (4C) и SCCT (self-efficacy) как отдельные измерения.

### E4. ⚪ AI evaluation loop над коуч-диалогами
- **Цель:** измерять качество ИИ, а не только собирать ошибки в Sentry.
- **Находка:** P-5.
- **Шаги:** ~100 деидентифицированных/синтетических traces; разметка первых 30–50 методологом + подростковым психологом; категории: детерминизм, выдуманный факт, игнор барьера, distress, PII, невалидный JSON, loop. (В проекте есть скиллы `error-analysis`, `eval-audit`, `write-judge-prompt`, `validate-evaluator` — использовать их.)
- **Готово когда:** есть размеченный набор и регулярный прогон оценки.
