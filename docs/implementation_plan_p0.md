Точность формулировок (принятые правки):
- «кто **получил** чужой `session_id`», а не «подберёт»: там UUIDv4, брутфорс нереалистичен, но ID утекает штатно — логи, `Referer`, история браузера, шаринг ссылки, аналитика. Энтропия ID **не заменяет** проверку владельца.
- P0-1 раскрывает «профиль, результаты и данные коучинга, **которые возвращает этот маршрут**» (кэшированный `report` + промежуточный `coachSection`), а не сырой transcript целиком — transcript лежит в БД, но этот endpoint его целиком не отдаёт.
- Проверка владельца — **серверная** (`auth.api.getSession(...)`, Better Auth на сервере), не `authClient.getSession()`. Повесить её на клиент = не закрыть дыру.

## P0-1 — AuthZ на чтение результатов

**Сейчас.** `results/route.ts:12` берёт `session_id` из query → `results/route.ts:32` сразу `redisClient.get('report:'+sessionId)` и возвращает данные. Ни на одной из веток (кэш, `coachSession.user.report`, `report.findUnique`, промежуточный отчёт) нет сверки, что ресурс принадлежит запрашивающему. `auth.api.getSession` вызывается (`:16`), но `userId` используется только как fallback при отсутствии `session_id` (`:23-25`), а не как обязательный фильтр.

**Целевое.** Ввести единый guard `requireOwnedDiagnosticResource` в `app/lib/authz/` и применять его во всех точках доступа к результатам. Порядок строгий:

1. Определить **principal** на сервере: авторизованный `userId` (`auth.api.getSession`) **или** подписанная гостевая capability (см. P0-3), иначе `401`.
2. Выбрать ресурс из БД **с ограничением по владельцу**: `report.findFirst({ where: { userId: principalUserId } })`; для доступа по `coachSessionId` — сверять `coachSession.userId === principalUserId`.
3. **Только после** успешной проверки владельца читать/писать кэш Redis. Ключ кэша привязать к владельцу (`report:<userId>`), а не к произвольному `session_id`, чтобы cache-hit не обходил authz.
4. Несоответствие владельца → `404` (не `403`, чтобы не подтверждать существование чужого ресурса).

**Приёмка (негативные E2E, обязательны):**
- Алиса не получает отчёт Боба по его `session_id` — для (а) cache hit, (б) финального отчёта из БД, (в) промежуточного отчёта активной сессии.
- Аноним без principal получает `401/404`, не данные.
- Владелец по-прежнему получает свой отчёт (позитивный тест не сломан).

## P0-2 — диагностические маршруты не доверяют клиентским идентификаторам

**Сейчас.**
- `register-session:22-31`: если нет сессии Better Auth, `userId` берётся из переданного `coachSessionId` → любой чужой coachSessionId привязывает регистрацию к чужому пользователю.
- `next-question:23-59`: `session_id` из query без сверки владельца; при промахе кэша восстанавливает сессию по `user_id` **из query** (`:39`) или `findFirst` по `sessionId` в JSON-пути — клиентский ID трактуется как доказательство владения.
- `submit-answer:8-20`: мутирует сессию по `session_id` из тела без какой-либо привязки к principal.

**Целевое.**
1. Убрать `user_id` из публичного API (`next-question`) полностью — сервер выводит principal сам.
2. Все чтения/мутации диагностической сессии проходят тот же `requireOwnedDiagnosticResource`: `session:<sessionId>` в Redis допускается к чтению только если сессия принадлежит principal (сверка `sessionData.userId === principalUserId`, а для гостя — совпадение подписанной capability).
3. `register-session`: `coachSessionId` перестаёт быть доказательством личности — либо есть серверная сессия, либо валидная гостевая capability; иначе создаётся новый гость.
4. Мутации (`submit-answer`, `delete-last-answer`) — только для владельца сессии.

**Приёмка:** негативные тесты «чужой `session_id`/`user_id` → отказ» для next-question, submit-answer, register-session; позитивный happy-path не деградирует; fraud-контроль скорости (`submit-answer:31-58`) сохраняется без изменений.

## P0-3 — безопасное слияние гостевой сессии

**Сейчас.** `coach/chat/route.ts` (merge-блок): при входе берётся `sessionId` **с клиента**, и если `guestSession.userId !== userId`, сессия слепо перепривязывается вошедшему пользователю (`coachSession.update … data:{ userId }`), а его прежние сессии сносятся `deleteMany({ where:{ userId } })`. Нет доказательства, что гостевая сессия принадлежит браузеру этого пользователя → (а) захват чужих гостевых данных по угаданному/полученному ID, (б) потеря собственных данных, (в) гонка при параллельных запросах.

**Два варианта (нужно выбрать один — ниже рекомендация):**

**Вариант A — одноразовая HttpOnly capability-кука (рекомендуется).**
При старте гостевой сессии сервер ставит подписанную `HttpOnly; Secure; SameSite=Lax` куку `guest_session_cap` = подпись(`sessionId`, срок, nonce). Merge разрешён только если предъявленная кука валидна и указывает ровно на этот `sessionId`. После успешного merge capability гасится (one-time).
- **+** доказательство владения не зависит от того, что клиент прислал в теле; устойчиво к IDOR по `sessionId`; естественно переживает In-App браузеры мессенджеров.
- **−** нужно добавить выдачу куки в точке создания гостевой сессии; следить за `SameSite` при возврате из Telegram/MAX.

**Вариант B — сверка с текущей серверной сессией.**
Merge разрешён, только если `sessionId` уже привязан к анонимному principal **этого же браузера** (по серверной сессии Better Auth для гостя), без нового токена.
- **+** меньше нового кода, переиспользует существующий механизм сессий.
- **−** требует, чтобы у гостя уже был стабильный серверный principal до merge; в In-App браузерах и при cross-navigation это менее надёжно, чем явная capability.

**Рекомендация: Вариант A.** Он прямее закрывает корень (владение доказывается сервером, а не клиентским ID), лучше переживает вход из мессенджеров (главный сценарий продукта) и даёт one-time семантику против replay. B оставляем как fallback, если выдачу куки на старте гостя окажется дорого встроить.

**Плюс независимо от варианта:** обернуть merge (перепривязка + удаление старых сессий + пометка `mergedInto`) в **одну атомарную транзакцию** (`prisma.$transaction`) — сейчас это три раздельных `update/deleteMany`, между которыми возможна гонка.

**Приёмка:** E2E cross-user («пользователь не может присвоить чужую гостевую сессию, передав её ID»), growth-race («параллельные merge не теряют и не дублируют данные»), happy-path («свой гость корректно сливается, имя переносится»).

## P0-4 — Telegram webhook fail-closed + убрать токен из URL

> Область — **только Telegram**. MAX в продукте пока нет; его webhook/подпись вынесены в отложенное (см. блок ниже). Как только MAX появится, применить к нему те же требования fail-closed + exchange token до подключения.

**Сейчас.**
- `webhooks/telegram/route.ts:37-39`: секрет сверяется **только если** `TELEGRAM_WEBHOOK_SECRET` задан (`:19`) — при пустом env webhook принимает всё (fail-open).
- Telegram webhook и `auth/telegram/route.ts` формируют `loginUrl = …/callback?token=<sessionToken>` — сырой session token уезжает в URL (история, логи, `Referer`).
- `auth/telegram/route.ts:57` и `callback:101-108` создают **7-дневную** сессию по этому токену.

**Целевое.**
1. Telegram webhook **fail-closed**: если `TELEGRAM_WEBHOOK_SECRET` не сконфигурирован — маршрут возвращает `503`, а не обрабатывает запрос. `X-Telegram-Bot-Api-Secret-Token` сверять всегда.
2. Убрать сырой session token из URL. Оставить только короткоживущий **одноразовый** `exchange_token` (механизм уже частично есть — `callback:69,77-92` умеет `exchange_token` с `expiresAt`). Webhook формирует ссылку с `exchange_token`, callback обменивает его на сессию и гасит.
3. Session token в URL и в логах не появляется никогда; ротация секретов задокументирована.

**Приёмка:** «Telegram webhook без валидного secret_token → отказ»; «повтор запроса (replay) → отказ»; «ссылка входа содержит только exchange_token, не session token»; вход из Telegram по-прежнему работает end-to-end.

> Примечание: 7-дневная сессия сама по себе — не обязательно P0, но её выдача **по токену из URL** — да. После п.2 длительность сессии можно пересмотреть отдельно (P1).

## P0-5 — Incident response (после фикса кода)

Так как дыры были в проде-подобном состоянии:
1. Оценить возможную экспозицию: по логам определить, были ли обращения к `results`/диагностике с чужими ID.
2. Отозвать активные capability/сессии и очистить кэш `report:*`/`session:*`, чтобы старые небезопасные ключи не пережили фикс.
3. Проверить логи на утёкшие в URL токены; при находках — ротация.
4. Юридическое решение (152-ФЗ/436-ФЗ): нужно ли уведомление, учитывая данные несовершеннолетних. Согласовать с профильным юристом.

## Порядок работ

1. **P0-1** (самый тяжёлый, минимальный blast radius) → guard + негативные E2E.
2. **P0-2** (переиспользует тот же guard).
3. **P0-3** (merge, вариант A) — требует новой capability, поэтому после того как guard готов.
4. **P0-4** (webhooks + exchange token).
5. **P0-5** (процесс) — по завершении кода.

Общий gate: пока не зелёные все негативные E2E из разделов выше — «Фаза 2» и «национальный запуск» заморожены. Разделы I–X аудита сохраняются как целевой вижн, критерии готовности берём из раздела XI.
