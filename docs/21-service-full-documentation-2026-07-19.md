## 1. Что это за продукт

Профориентационная веб-платформа для подростков 13-17 лет и их родителей.
Путь пользователя: живой диалог с ИИ-коучем «Роман» → батарея из 8
психометрических тестов → персональный отчёт с рекомендациями профессий,
картой сильных сторон и планом первых шагов. Домен: `synthosai.ru`.

Ценностное предложение — не «тест угадывает профессию», а триангуляция трёх
источников (коуч-диалог, тесты, объективная согласованность между ними) в
единый цифровой профиль подростка.

## 3. Модель данных (Prisma, 11 моделей)

```
User (STUDENT|PARENT)
 ├─ Session[] / Account[]           — Better Auth
 ├─ CoachSession (1:1)              — вся коуч-сессия целиком
 ├─ DiagnosticResult[] (1:N)        — по одной записи на testCode
 ├─ DigitalProfile (1:1)            — 8-слойный агрегированный профиль
 ├─ Report (1:1)                    — сгенерированный JSON-отчёт
 ├─ AuthLink[]                      — QR/код подключения мессенджера
 └─ child/parents (self-relation)   — связь родитель-ребёнок (задел на будущее)
```

Ключевые нюансы, важные для понимания системы:

- **`CoachSession.version`** — optimistic locking (число, инкрементируется на
  каждую запись). Защищает от гонки при двойном клике/ретрае сети, которая
  раньше могла молча откатывать прогресс Пирамиды Идентичности.
- **`CoachSession.transcript`** — не массив, а объект `{ EXPRESS: [...],
  DEEP: [...] }`: две независимые истории диалога для двух режимов сессии.
- **`CoachSession.extractedData`** — плоский JSON с полями `expressExtracted`
  и `deepExtracted` внутри (не путать с `DigitalProfile.summary` — это разные
  структуры на разных стадиях конвейера, см. §6 и §7).
- **`DiagnosticResult`** — при каждом прохождении тестов таблица полностью
  перезаписывается (`deleteMany` → `createMany`) для юзера, история
  предыдущих попыток не хранится.
- **`DigitalProfile.summary`** — единственное место, куда сведены **и** коуч,
  **и** тесты в общую 8-слойную схему (`app/lib/profile/layers.ts`). До
  19.07.2026 эта таблица писалась, но нигде не читалась (см. §7, §11).
- Гостевые пользователи создаются с `email: guest_{random}@moiprizvanie.ru`;
  при авторизации выполняется **soft merge** (`mergedInto`), не физическое
  удаление — исторические данные гостя не теряются.

## 4. Путь пользователя (end-to-end)

```
Лендинг (/) 
  → Коуч EXPRESS (/coach, режим по умолчанию, шаги 0-16)
     [опционально] → Коуч DEEP (тот же /coach, шаги 16-22, отдельный transcript/extractedData)
  → Тесты (/assessment, 8 модулей: RIASEC → BFI → ICAR → VIA → PROCRASTINATION → PVQ → GROWTH → CONTEXT)
  → Отчёт (/report) — ИИ-генерация + детерминированные надстройки (архетип, coverage, консистентность)
  → Личный кабинет (там же /report, CabinetProgress) — статус 3 этапов, ссылки на PDF/повтор
```

Авторизация не блокирует старт — гость может пройти коучинг без «регистрации»
(слово запрещено в UI, см. `docs/03-auth-policy.md`); подключение
Telegram/MAX ID происходит на шаге 2 диалога, органично внутри разговора, не
как отдельная форма.

входит в зону ответственности кода, не решается коммитом.

## 6. Коуч-сессия — детальная логика

### 6.1 Методология

Согласно `docs/02-neurocoach-methodology.md` и промптам (`app/lib/coach/prompts.ts`):
- **CLEAR** (Хокинс) — контракт, слушание, исследование — задаёт тон EXPRESS-этапа.
- **WOOP** (Эттинген) — желание/результат для мечт и целей.
- **Solution-Focused Brief Coaching** — фокус на будущем, не копание в травмах.
- **Пирамида Дилтса (адаптированная)** — используется как драматургия DEEP-сессии
  (Запрос → Результат → Эмоции → Идентичность → Действия → Первый шаг), но
  **с 19.07.2026 переформулирована под подростковый язык**: шаги «Эмоции» и
  «Идентичность» больше не задают абстрактные вопросы напрямую («что ты
  чувствуешь?», «кто ты?»), а используют проективные/телесные образы —
  решение по итогам сегодняшнего аудита, ключевая причина изначальной жалобы
  «коуч крутит 2-3 уровень пирамиды».
- **«Адвокат дьявола»** (Д-4) — мягкое одноразовое оспаривание шаблонных
  ответов подростка, guard от повторного применения на одном шаге.

### 6.2 Два режима: EXPRESS и DEEP

| | EXPRESS | DEEP |
|---|---|---|
| Шкала шагов | 0-16 | 16-22 (шаги 0-15 общие с EXPRESS) |
| Transcript | `CoachSession.transcript.EXPRESS` | `CoachSession.transcript.DEEP` |
| Данные | `extractedData.expressExtracted` | `extractedData.deepExtracted` |
| Финал | Резюме + приглашение к тестам | Резюме + `deepReportSummary` (отдельный синтез для отчёта) |
| Ratchet прогресса | `maxStepReachedByMode.EXPRESS` | `maxStepReachedByMode.DEEP` — раздельно, не откатывается назад |

Обе шкалы **унифицированы на едином номерном пространстве** (0-22) — до
исправления в прошлом аудите DEEP использовал отдельную шкалу 10-16, что
приводило к рассинхрону шапки/степпера и «мёртвому» коду анти-залипания.

### 6.3 Экстракция данных

На каждом ходу диалога отдельный вызов LLM (`generateJson`, температура 0.1)
извлекает поля, релевантные текущему шагу (см. таблицу схемы в
`chat/route.ts`). Есть fallback-экстрактор на regex (`fallbackExtract`) —
подстраховка, если LLM не вернул валидный JSON.

**Анти-залипание:** если `nextStep` не меняется 2 хода подряд (`stepAttempts
>= 2`), поле форс-заполняется каноническим текстом, и шаг принудительно
продвигается — не даёт разговору зависнуть на одном вопросе бесконечно.

### 6.4 Известные технические решения этой сессии (19.07.2026)

1. Починен путь к данным в финальном промпте DEEP (`extractedData.deepGoal`
   → `extractedData.deepExtracted.deepGoal`) — раньше финальная реплика
   всегда была обезличенной (пустые строки).
2. Убран осколок старой шкалы (`currentStep: 10` в приветствии DEEP).
3. Коуч больше не угадывает архетип по Юнгу сам в финалке (см. §8).

## 7. Диагностическая батарея

8 модулей (`app/data/questions.ts`, `app/lib/diagnostic/scoring.ts`),
обёрнуты в «квест»-нарратив (`app/data/narrative.ts` — глава/интро/аутро/эмодзи
на каждый блок, без сюжетных развилок — вопросы остаются шкалой 1-5).

| testCode | Методика | Что измеряет | Добавлено |
|---|---|---|---|
| RIASEC | Голланд | 6 типов интересов + Код Холланда | исходно |
| BFI | Big Five | O/C/E/A/N + LOC + AMB | исходно |
| ICAR | International Cognitive Ability Resource | verbal/numeric/spatial, полоса относительно возраста | исходно |
| PROCRASTINATION | Шкала Лэя | уровень избегания задач | исходно |
| VIA | VIA Youth Survey | 24 силы характера → 5 добродетелей + Трансцендентность (агрегаты) | добродетели починены 19.07 |
| PVQ | Портретный опросник ценностей Шварца | 10 ценностей, ипсатизация, топ-3 | исходно |
| **GROWTH** | Адаптации Grit-S / Growth Mindset / TEIQue-SF | настойчивость, установка на рост, эмоц. интеллект | **новое, 19.07.2026** |
| **CONTEXT** | Авторский короткий самоотчёт | семья, здоровье, мобильность, среда, готовность выбирать | **новое, 19.07.2026** |
| **COGNITIVE_STYLE** | Собственные пункты (не лицензированный BRIEF/MSLQ) | тормозной контроль, гибкость, стиль обучения, самоэффективность, метакогниция, любознательность | **запланировано, `docs/20` §8 Фаза 4b** |

Причина появления GROWTH/CONTEXT: методологический аудит (`характеристики для
сбора.doc/.pages`) прямо критиковал попытку коуча оценивать эти же
конструкты «на глазок» по диалогу как низковалидную практику — тесты
заменили эвристику ИИ на короткие, но настоящие измерительные шкалы.

**Скоринг** — единый реестр `scorers: Record<string, TestScorer>`
(`scoring.ts`), каждый скорер — чистая функция `(answers, questions,
context?) → { scores, reliability }`. Добавление нового testCode требует
правки в 5 местах: `questions.ts` (вопросы), `scoring.ts` (скорер +
регистрация), `narrative.ts` (глава квеста + порядок), `CabinetProgress.tsx`
(бейдж главы), сборка в `next-question/route.ts` (маппинг в профиль).

## 8. Цифровой профиль (`DigitalProfile.summary`)

8-слойная схема (`app/lib/profile/layers.ts`): Интересы, Личность, Сильные
стороны, Когнитивный профиль, Мотивация, Социальное взаимодействие,
Поведение, Контекст + слой согласованности (`consistency`) + `coverage`
(доля заполненности 0-1 по каждому слою, `computeCoverage()`).

**Ключевая находка аудита 19.07.2026:** эта структура честно считалась и
сохранялась в БД, но **нигде не отображалась пользователю** — write-only
слой. Исправлено: `coverage`, тест GROWTH и тест CONTEXT теперь выводятся в
`/report` тремя блоками через переиспользуемый `ValueBars`.

Из ~100-120 характеристик исходного методологического документа (полная
сверка — `docs/19`):
- **~55-60% активно собираются и показываются** в отчёте.
- **~10%** собираются, но пока не отрисованы (низкий приоритет UI).
- **~24%** — конкретный, оценённый, но ещё не реализованный план (`docs/20`,
  §8: DERIVE-классификатор, расширение GROWTH, новый тест COGNITIVE_STYLE —
  решение от 19.07.2026 делать когнитивный кластер собственными пунктами, не
  ждать лицензированный инструмент).
- **~25%** — осознанно не собираются (дубли полей либо конструкты, которые
  сам аудит просил исключить как «взрослые»/рискованные).
- **2 поля** — принципиально не автоматизируются без психолога (буллинг,
  зависимость от сверстников).

## 9. Отчёт (`/report`)

Строится в `app/api/v1/diagnostic/next-question/route.ts` в момент, когда
отвечен последний вопрос батареи:

1. Скоринг всех 8 тестов через единый реестр `scorers`.
2. Детерминированный расчёт **архетипа** (`deriveArchetype()`, VIA+PVQ,
   взвешенная модель Пирсон-Марр — см. §10 про статус этого блока).
3. Детерминированный расчёт **триангуляции** (`computeConsistency()`) —
   сравнение тестовых фактов с фактами коуча, формулирует «что сошлось / что
   разошлось» как точку роста, не как ошибку.
4. Сборка `DigitalProfile.summary` (`buildSummaryProfile()`).
5. Один вызов LLM (`generateJson`, температура 0.7) — синтез нарративного
   текста отчёта (резюме, сильные стороны, профессии) по строгой JSON-схеме;
   при сбое — детерминированный fallback-отчёт на реальных баллах (не пустой
   экран).
6. Результат кладётся в `Report.htmlContent` (несмотря на название поля —
   это JSON, не HTML) и кэшируется в Redis на 24 часа.

`/report` (фронтенд) читает `Report.htmlContent` через
`/api/v1/diagnostic/results`, санитизирует и рендерит по вкладкам
(Личность/Карьера/Коуч-сессия).

## 10. Архетип — методологическое решение (обновлено 19.07.2026)

Собственный Юнгианский тест архетипов был методологически слабым местом
(эффект Барнума, низкая психометрическая валидность). Решение, принятое
сегодня — **не убирать полностью, а честно переформулировать**:

- Расчёт остаётся детерминированным (`app/lib/profile/archetype.ts`):
  взвешенная комбинация относительной выраженности 6 добродетелей VIA и 10
  ценностей PVQ, без угадывания LLM.
- В отчёте карточка называется **«Твоя метафора роли»**, а не «Твой
  архетип» / «Твой тип личности», с явной припиской: *«образный ярлык
  поверх твоих результатов VIA и ценностей — не психологический тип, а
  способ увидеть себя в истории»*.
- Коуч в финалке чата (EXPRESS и DEEP) **больше не угадывает архетип сам** —
  раньше LLM называл архетип вживую в чате, до того как проходили тесты,
  что могло противоречить более позднему детерминированному результату.

## 11. UI/UX-система

### 11.1 Темы
Тёмная (по умолчанию) — «неоновый космос»: `--accent-brown: #3B82F6`
(на самом деле синий, имя переменной устарело), `--accent-gold: #00F0FF`
(неоновый циан), фон `#040506`. Светлая — «капучино/латте»: тёплые бежевые
и коричневые тона (`#FDFBF7`…`#3E2723`). Переключение через
`document.documentElement[data-theme]`, наблюдается `MutationObserver`'ом в
компонентах (`PyramidOfAlignment` и др.), не через React-контекст.

### 11.2 Цветовое кодирование склонностей (RIASEC/таланты)
Отдельная палитра `--riasec-*` (creative, tech, analytical, social,
organizational, startup + outcome/goal для пирамиды) — используется
одновременно в `WheelOfVocation` (Колесо талантов) и `PyramidOfAlignment`
(Пирамида Идентичности) для визуальной консистентности.

### 11.3 «Квест»-обёртка тестов
`narrative.ts` — лёгкая нарративная обёртка (не полноценный VN-движок):
заголовок главы, эмодзи, интро/аутро-текст. Реализовано и работает с исходной
версии продукта; сегодня подтверждено как правильный баланс «геймификации
без риска для валидности тестов» (в противовес отклонённому VN-blueprint,
см. §12).

### 11.4 Ключевые визуальные компоненты
`PyramidOfAlignment.tsx` (DEEP-прогресс, 6 уровней SVG-полигонов),
`WheelOfVocation.tsx` (радар склонностей), `ValueBars.tsx` (переиспользуемый
горизонтальный bar-chart — сегодня использован для 3 новых блоков отчёта),
`CabinetProgress.tsx` (3-этапный прогресс ЛК).

### 11.5 Технический долг UI (из прошлого аудита, не тронуто сегодня)
~500 захардкоженных hex-цветов в 20+ TSX-файлах вместо CSS-переменных;
светлая тема местами держится на point-фиксах в `globals.css`, а не на
токенах. Mobile Speed Index 8.9s из-за тяжёлого анимированного фона.

## 11.6 База отраслей и профессий (`docs/industries_and_professions/`)

Отдельный документ-концепт (20 отраслей, 119 профессий) — проверен на
соответствие коду 19.07.2026. Картина та же, что и с VN-blueprint: часть
реально работает, часть — мёртвая документация, не подключённая к системе.

**Реально работает:**
- `app/data/professions_db.ts` (2241 строк) — 119 профессий, 20 отраслей, с
  RIASEC/Gardner/Big Five профилем, предметами ЕГЭ, формулой навыков.
  Используется в `RpgProfessionCard.tsx` (карточка профессии в отчёте).
- **Механизм подбора:** профессии в отчёте **на 100% свободно называет LLM**
  в момент генерации (`next-question/route.ts`, промпт «Предложите 3-5
  конкретных профессий»). `professionsDb` подключается только **постфактум**
  как обогащение — `RpgProfessionCard` ищет точное совпадение по имени
  строки в `professionsDb.find(p => p.name === name)`; если LLM назвал
  профессию, которой нет в базе (или назвал чуть иначе), обогащения не
  происходит, карточка остаётся с голым текстом LLM.

**Не работает / мёртвый код:**
- `app/data/characteristics_mapping.ts` (655 строк, primary/secondary/anti
  веса по каждой профессии) — импортируется только из `characteristics_db.ts`,
  который **не импортируется вообще нигде**. Плюс использует коды полей
  (`riasec_conventional`, `cog_math_logic`, `exec_inhibition`,
  `belbin_doer`…), которых **не существует** в реальной схеме
  `app/lib/profile/layers.ts` — тот же класс проблемы, что был у
  VN-blueprint: документ сгенерирован независимо и не сверен с фактическим
  кодом.
- Весь `integration_concept_plan.md`: детерминированный скоринг-модуль
  (`app/lib/diagnostic/matching.ts`), фронтенд-компонент
  `InteractiveJobBadge`, 3D-карта «Созвездия Отраслей» (Three.js), «Odyssey
  Plans» (3 траектории), трекер микро-проб — **ни один файл не существует**.

**Рекомендация (тот же принцип, что и по VN-blueprint — валидность и
реализуемость важнее визуального эффекта):**

1. **Высокая ценность, низкий риск:** заменить свободное угадывание профессий
   LLM на **реальный детерминированный шорт-лист** — простая взвешенная
   функция совпадения `DigitalProfile.summary` (RIASEC, Big Five, VIA — поля,
   которые реально существуют) против `professionsDb`, топ-8-10 кандидатов
   передаются в промпт LLM как **готовый список на выбор** вместо полностью
   открытой генерации. Это устраняет проблему несовпадения имён (обогащение
   работает всегда, не только «если повезло») и снижает риск, что LLM
   насочиняет профессию не по данным подростка. Оценка: 0.5-1 день, без
   новых таблиц БД и UI.
2. **Средняя ценность:** `InteractiveJobBadge` в чате коуча (профессии в
   обратных кавычках → кликабельные карточки) — приятная деталь, не срочно.
3. **Отложить:** 3D-карта «Созвездия», Odyssey Plans, трекер микро-проб —
   тот же диагноз, что у VN-blueprint: большой объём (Three.js-движок, новые
   таблицы БД для трекера) без проверенной необходимости. Делать только
   после того, как базовый детерминированный мэтчинг (п.1) подтвердит
   ценность на реальных пользователях.
4. **Housekeeping:** `characteristics_mapping.ts` и `characteristics_db.ts`
   либо переписать на реальные имена полей из `layers.ts` (если решаем
   делать п.1 через них), либо удалить как неиспользуемый, вводящий в
   заблуждение код — на выбор владельца.

## 12. VN/RPG-blueprint — рассмотрен и отклонён

Внешний документ (`Antigravity IDE`, не наш код) предлагал полную
перестройку на визуальную новеллу + RPG-карту + систему архетипов с «Тенью».
Разобран подробно в диалоге 19.07.2026, решение — **не строить**:
психометрика ломается сюжетными развилками, объём (150 характеристик / 75
минут) противоречит найденной сегодня усталости подростков на текущих ~21
шаге, «Святилище Архетипов» с shadow-work для несовершеннолетних требует
участия психолога, которого в плане не было. Полный разбор и контрпредложение
— `docs/20-blueprint-my-proposal-2026-07-19.md`.

## 13. ИТОГО: что оставляем / что убираем / что меняем — и почему

### Оставляем (работает, не трогаем)

| Что | Почему |
|---|---|
| Архитектура данных: `CoachSession` + `DiagnosticResult` + `DigitalProfile` + `Report`, без новых параллельных таблиц | Единый источник данных проще поддерживать; сегодняшняя путаница между `extractedData` и `DigitalProfile.summary` — аргумент **против**, а не за добавление ещё одного хранилища |
| CLEAR / WOOP / Solution-Focused как основа методологии коуча | Признано соответствующим мировым практикам методологическим аудитом |
| 8 психометрических тестов как Likert-шкалы 1-5 | Валидность инструмента важнее сюжетного оформления |
| Лёгкая «квест»-обёртка тестов (`narrative.ts`) | Даёт геймификацию без риска для валидности — правильный баланс, подтверждённый сегодняшним разбором VN-blueprint |
| Детерминированный расчёт архетипа и триангуляции (не LLM-угадывание) | Методологически валиднее и воспроизводимее |
| Soft-merge гостевых аккаунтов, optimistic locking `CoachSession.version` | Защищает данные пользователя от гонок и потерь |

### Убираем / убрали

| Что | Почему |
|---|---|
| Угадывание архетипа коучем в финалке чата | Дублировало и могло противоречить детерминированному расчёту в отчёте |
| Ярлык «Твой архетип» / «тип личности» | Риск восприниматься как психологический диагноз (эффект Барнума) — заменено на явно подписанную метафору |
| Оценка Grit/Mindset/TEIQue/семейного контекста «на глазок» коучем | Низкая психометрическая валидность — заменено на короткие тесты GROWTH/CONTEXT |
| ~25 полей схемы профиля (дубли или «взрослые»/рискованные конструкты: `stressEvaluation`, `digitalHygiene`, `socialCapital`, `parentalInfluence` и др.) | Дублируют другие поля либо прямо попадают под критику методологического аудита («корпоративные»/неуместные для возраста конструкты) — план удаления в `docs/20` §8, Фаза 5 |
| VN/RPG-blueprint целиком (сюжетные развилки, Cyberpunk/Anime/Minimal скины, Canvas-карта, «Святилище Архетипов», 4 новые таблицы БД) | Ломает валидность тестов, утраивает объём сессии, вводит клинически рискованный контент без психолога, требует месяцы работы вместо развития рабочей системы |
