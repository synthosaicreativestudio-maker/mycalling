## 1. 🔴 Живой баг: задвоение вопросов коуча (root cause)

### Симптом (скриншот 24.07, прод, новая сессия гостя)
1. Коуч: «Приятно познакомиться! Теперь давай подключим канал связи…» (шаг 2, `!hasPhone`)
2. Коуч: «Спасибо за выбор канала! 😉 Роман, сколько тебе лет?» (авто-переход)
3. Пользователь: «15 лет»
4. Коуч: **повторяет** «…давай подключим канал связи…» ← регрессия шага

### Корневая причина (архитектурная)
Машина состояний **не хранит номер шага как монотонный счётчик**, а **пересчитывает его
заново на каждый запрос** из «мягких» булевых флагов:

```
route.ts:379-384   hasName / hasPhone / hasAge / hasGrade / hasCity / hasPersonalInfo
route.ts:406-421   psychoBlocksCount (15 полей, извлекаются ИИ)
route.ts:533       nextStep = !hasName ? 1 : (!hasPhone ? 2 : (!hasPersonalInfo ? 2 : ...))
```

Ключевой флаг — `hasPhone`:
```
route.ts:380   hasPhone = !!coachSession.user.phone || !!extractedData.phone
```
`extractedData.phone` выставляется **только** при подтверждении через Telegram-бота:
```
route.ts:741-746   if (isPhoneConfirmedMsg) { extractedData.phone = dbUser.phone }
```
**У гостя без привязанного Telegram `phone` нестабилен.** Авто-переход к вопросу возраста
(`route.ts:641`) срабатывает, когда `hasPhone` на миг `true`. Но на следующем реальном
сообщении («15 лет») `hasPhone` вычисляется снова — и если он `false`, вся машина
откатывается на `nextStep = 2` → снова печатает приглашение подключить канал = **задвоение**.

**Вывод:** любой флаг, который «мигает» (особенно `hasPhone` у гостей и `psychoBlocksCount`
от ИИ-экстрактора, который иногда не извлекает поле), двигает шаг **назад** и повторяет
вопрос. Это не регрессия 22 июля — это свойство дизайна 20 июля.

### Почему костыли не спасают
В коде уже есть «anti-stuck» заплатки (`route.ts:536-630`, `stepAttempts`), авто-переход
(`route.ts:634-665`) и `fallbackExtract`. Они борются со *следствиями*, добавляя ещё
состояний, но не устраняют *причину* — вывод шага из нестабильных флагов.

### Правильное лечение (не откат)
- Ввести **монотонный серверный счётчик шага**: шаг может только расти; флаги лишь
  разрешают/запрещают переход вперёд, но **никогда не двигают назад**.
- Отделить статус авторизации (`hasPhone`) от прогресса анкеты: гость проходит анкету
  без «телефона»; привязка канала — это отдельный опциональный слой, не гейт шага.
- Детерминированные шаги 0–2 — строго по шаблонам, без ИИ (уже зафиксировано инвариантом
  в `.agents/AGENTS.md`, но код это правило не соблюдает для `hasPhone`).

## 2. 🔴 Безопасность: RLS выключен

- Единственная миграция `prisma/migrations/20260626051706_init` **не содержит** ни одной
  `ENABLE ROW LEVEL SECURITY` / `CREATE POLICY` (проверено grep — 0 совпадений).
- Таблицы `users`, `coach_sessions`, `diagnostic_results`, `digital_profiles`, `reports`
  хранят **персональные данные несовершеннолетних** (имя, возраст, класс, город, глубокий
  психологический профиль) — и **не защищены RLS**.
- Prisma ходит под service-role, поэтому *приложение* работает; но при утечке anon-ключа
  или ошибке в API любой сможет читать чужие строки.
- **Действие:** подготовить SQL-миграцию RLS (политика «пользователь видит только свои
  строки», серверные операции — под service-role). Применяет владелец. **До публичного
  запуска.**

## 4. 🟡 Отчёт: экран ≠ PDF (дублирующаяся вёрстка)

- `report/page.tsx` держит **две вёрстки**: живую (`print:hidden`, строка 478) и ручную
  печатную (`hidden print-container`, строка 1091) + `@media print` (строка 454).
- Любая новая секция, добавленная в экранную вёрстку, **не попадает в PDF автоматически** —
  их надо синхронизировать руками. Это источник «部分 секций теряется в печати».
- **Действие:** единый источник секций + грамотный `@media print`, либо тест-инвариант
  «каждая секция имеет представление в печати».
