# 31 — План: надёжность отчёта, полный учёт ответов в матчинге, Колесо+Пирамида на /report, Bento-сетка

**Дата:** 2026-07-24
**Статус:** план, выполняется пошагово в этом же прогоне.
**Триггер:** пользователь на `/report` увидел (а) голый абзац RIASEC вместо интегрированного нарратива, (б) отсутствие Колеса талантов и Пирамиды идентичности, (в) вертикальную «портянку» вместо плиточной раскладки. Плюс явное требование: рекомендации профессий должны учитывать **каждый ответ** и **верифицироваться по цифровому паспорту профессии**.

---

## Находки (подтверждено чтением кода, не догадки)

1. **`coachData` в `next-question/route.ts` читает только 4 текстовых поля коуча** (`dreams`, `idols`, `values`, `fears/barriers` + 6 deep-полей) — хотя `getField()` умеет доставать любое поле. **10 полей экспресс-коучинга полностью игнорируются**: `hobbies`, `schoolSubjects`, `experience`, `workFormat`, `thinkingType`, `successMeasure`, `energySources`, `teamRole`, `autonomyStyle`, `decisionStyle`. Они не попадают ни в промпт для ИИ-нарратива, ни в детерминированный матчинг.
2. **`matchProfile` (вход в `topArchetypes`) вообще не видит текстовые ответы коуча** — только `riasec`, `bigFive`, `topValues` (PVQ), `signatureStrengths` (VIA), `icarBand`. Пять осей паспорта профессии (`riasec`, `bigFive.traits`, `values`, `viaFit`, `cognitiveDemand`) сверяются с тестами, но **не с тем, что подросток рассказал коучу**. Это и есть корень «не учитывает каждый ответ».
3. **Паспорт профессии (`professions_db.ts`) уже имеет 7 полей**, из них `gardner` и `subjects` **не участвуют в `computeFit`** вообще (только riasec/bigFive/values/viaFit/cognitiveDemand — 5 из 7 возможных осей).
4. **Отчёт `/report` никогда не импортировал `WheelOfVocation`/`PyramidOfAlignment`** (`grep` не находит ни одного упоминания) — они существуют только на `/coach`. Не сегодняшняя регрессия — никогда не было сделано.
5. **Вкладка «Карта талантов» — вертикальный список карточек**, не грид/Bento (Stage 13 бэклога не доведён либо откатился).
6. **Надёжность генерации:** `generateJson` при сбое Gemini (тот же нестабильный провайдер, что тормозил коуча) откатывается на **generic-фолбэк** с одной строкой RIASEC-нарратива (`riasecSummary` строится по шаблону "Ваши ключевые интересы определены по методике RIASEC. Наибольшую активность проявляют сферы: X и Y.") — то, что увидел пользователь. Профессии в фолбэке **детерминированно строятся** (`buildReportProfessions(null)`), значит список профессий должен быть непустым даже при сбое ИИ — если пользователь видел пустую вкладку, вероятная причина — **устаревший кэш в Redis/БД** (TTL 24ч) от более раннего прогона, не текущий код.

## Блок A — Надёжность генерации отчёта (первым, самое важное)

**Цель.** Сбой Gemini не должен превращать отчёт в куцый абзац. Фолбэк обязан быть содержательным и опираться на реальные данные (тесты + коуч), а не на один шаблон.

### A1. Расширить `coachData` до всех 10 полей экспресс-коучинга
Добавить в объект `coachData` (`next-question/route.ts`): `hobbies`, `schoolSubjects`, `experience`, `workFormat`, `thinkingType`, `successMeasure`, `energySources`, `teamRole`, `autonomyStyle`, `decisionStyle`. Прокинуть их в `coachDataPrompt` (для ИИ-нарратива) — сейчас туда попадают только 4 поля.

### A2. Обогатить фолбэк-отчёт качественными данными коуча
Сейчас `fallbackReport.riasecSummary` — один шаблонный абзац. Добавить деривацию: если `coachData.hobbies`/`dreams` непусты — вставить их текстом в `heroSummary`/`strengths` фолбэка (детерминированно, без ИИ), чтобы даже при сбое Gemini отчёт был персонализирован, а не универсален.

### A3. Тайм-аут и повтор для генерации отчёта
Текущий `callFreemodelWithRetry(requestBody, 25000)` — 25с, 1 повтор на ключ. Учитывая нестабильность провайдера (тот же корень, что и с коучем), поднять до 2 попыток на ключ для отчёта конкретно (не глобально — это разовая операция при завершении теста, не чат) и/или использовать `EXTRACTION_MODEL`-паттерн: если задан `env.EXTRACTION_MODEL`, не трогаем (это для коуча); для отчёта — отдельный анализ не требуется, полагаемся на A1+A2 (фолбэк, а не бесконечные ретраи).

### A4. Инвалидация кэша при новом прогоне
Убедиться, что `next-question/route.ts` при завершении теста **перезаписывает** Redis-кэш (`EX 86400`) и Report-строку в БД свежим содержимым (уже `upsert`, подтверждено) — то есть A1-A2 автоматически видны при следующем реальном прохождении теста. Отдельный код не нужен, но важно предупредить владельца: **старые кэшированные/уже сохранённые отчёты не обновятся сами** — нужен новый прогон теста для проверки.

**Критерий.** Фолбэк-отчёт содержит хотя бы 2 персонализированных предложения из данных коуча, даже при полном сбое Gemini.

## Блок B — Полный учёт ответов в матчинге + верификация по паспорту

**Цель.** Ранжирование профессий учитывает **все** структурированные тесты **и** текстовые ответы коуча, а не только 5 числовых осей.

### B1. Текстовый бонус по паспорту (симметрично `talentSignals.ts` из коуча)
Профессия уже имеет `subjects: string[]` (предметы/склонности) — паспорт. У ученика есть текстовые `hobbies`, `schoolSubjects`, `experience` (из A1). Добавить `subjectsFit`: пересечение слов из текста ученика с `profession.subjects` (по совпадению подстрок/лемм, как в `talentSignals.ts`) → доля совпадения 0–1. Это **прямая верификация профессии по тому, что подросток реально написал**, а не только по тестовым баллам.

### B2. Ось Гарднера (`gardner`)
У профессии есть `gardner: (...)[]`. Вывести интеллекты Гарднера из тех же источников, что и RIASEC/talentScores (сигналы из ответов коуча +, опционально, соответствие типовым RIASEC→Gardner маппингам). Добавить `gardnerFit` в `computeFit` по образцу `overlapFit`.

### B3. Обновить веса осей с ре-нормировкой
Текущие веса (interests .30, personality .15, values .15, strengths .15, cognitive .10 = .85). Добавить `subjects: .10`, `gardner: .05` → сумма 1.00. Ре-нормировка уже реализована (`weightSum`), новые оси корректно деградируют при отсутствии данных.

### B4. `MatchProfile` расширить текстовыми полями
Добавить в интерфейс `MatchProfile`: `textSignals?: { hobbies, schoolSubjects, experience }` (сырые строки для B1). Прокинуть из `next-question/route.ts`'s `matchProfile` объекта (используя `coachData` после A1).

### B5. Golden-тест на регрессию ранжирования
Тест «профиль → топ-5» (как рекомендовано в исходном docs/25 плане) — чтобы будущие правки весов были осознанными, не тихими.

**Критерий.** У профессии с явным совпадением по `subjects`/`gardner` итоговый score заметно выше, чем без; `npx vitest run` зелёный; экспресс-путь (без части осей) не падает.

## Блок C — Колесо талантов + Пирамида идентичности на `/report`

**Цель.** Финальный отчёт визуально показывает то же Колесо и Пирамиду, что и коуч-сессия — не только текстовый нарратив.

### C1. Прокинуть `talentScores` и `deepReportSummary` в JSON отчёта
Уже частично есть (`report.professions`, `report.deepSession`) — добавить `report.talentScores` (7 шкал из `expressExtracted.talentScores`) в JSON отчёта (`next-question/route.ts`, оба пути: успех и фолбэк).

### C2. Рендер `WheelOfVocation` на вкладке «Карта талантов»
Импортировать компонент в `report/page.tsx`, адаптировать пропсы (сейчас он читает `talentScores` через хук/пропс на `/coach` — проверить сигнатуру и переиспользовать тот же компонент в read-only режиме, без чата).

### C3. Рендер `PyramidOfAlignment` (если пройдена глубокая сессия)
Аналогично — показывать Пирамиду с данными `deepReportSummary`, только если `coachSessionMode==='DEEP'`/`deepSessionCompletedAt` есть.

**Критерий.** На `/report`, вкладка «Карта талантов», видно то же Колесо (с реальными баллами) и, если пройдена глубокая сессия, Пирамиду.

## Блок D — Bento-сетка вместо «портянки»

**Цель.** Вкладка «Карта талантов» — плиточная раскладка (грид), не вертикальный список.

### D1. Инвентаризация текущих карточек вкладки
Собрать список блоков, которые сейчас рендерятся вертикально (Hero, Big Five, RIASEC-радар, VIA, Колесо/Пирамида из C, увлечения/анти-интересы, Савикас, прокрастинация).

### D2. CSS-grid раскладка (Bento)
Перевести контейнер в `grid` с разноразмерными ячейками (`grid-template-areas` или auto-flow dense) — крупные карточки (Колесо, Пирамида) на 2 колонки, мелкие (отдельные метрики) на 1. Уважать mobile (одна колонка на узких экранах).

**Критерий.** Проверяет владелец визуально (десктоп + мобиль, обе темы) — это структурная UI-правка, риск по [[feedback_careful-layout-edits]].
