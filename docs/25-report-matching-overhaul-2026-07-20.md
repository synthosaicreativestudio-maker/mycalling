## 0. Что выяснено при исследовании (факты, не догадки)

### 0.1. Матчинг использует ~30 % собранных данных
`app/lib/profile/professionMatch.ts` считает совпадение так:
```
fit = 0.7 · riasecFit + 0.3 · bigFiveFit
```
Больше **ничего** не участвует. При этом:
- Паспорт профессии (`app/data/professions_db.ts`) уже содержит оси
  `values` (PVQ), `viaFit` (VIA), `cognitiveDemand`, `gardner`, `subjects`,
  `skills` — и в самом интерфейсе они помечены как «оси многомерного матчинга,
  по ним считается совпадение сверх RIASEC».
- Профиль ученика (`app/lib/profile/layers.ts`, `SummaryProfile`) собирает 8 слоёв
  (~120 характеристик): RIASEC, Big Five (+ LOC, толерантность к неопределённости,
  grit, TEIQue, mindset), VIA (24 силы + 6 добродетелей + signatureStrengths),
  ICAR (по подшкалам + band), PVQ (topValues), Савикас (4 ресурса),
  прокрастинация, Belbin-роли, контекст/ресурсы.
- **Всё это хранится в `digital_profiles.summary`, но матчер его не читает.**

**Вывод:** многомерный матчинг был спроектирован (паспорта + профиль), но не
подключён. Это главный методологический разрыв. Мы обещаем «подбор по всем
характеристикам», а фактически подбираем по интересам + личности.

### 0.2. PDF отстал от живой страницы
`app/report/page.tsx` рендерит **две независимые вёрстки**:
- живую (`<main class="print:hidden">`) — вся портянка секций;
- печатную (`<div class="hidden print-container">`, строки ~1037–1120) —
  ручной дубль для `window.print()`.

Печатная версия **не содержит**: детали Big Five, «Карьерная адаптивность
(Савикас)», «Склонность к прокрастинации», «Увлечения и анти-интересы»,
`profileCoverage`. Их добавили в живую страницу и забыли перенести в печать.
Коуч-секция и глубокая сессия в печать **попадают** (если заполнены).

### 0.3. Глубокая коуч-сессия — есть, но спрятана
Точка входа одна: `app/assessment/page.tsx:212` — предложение показывается только
если `coachSessionMode === 'EXPRESS' && !deepSessionCompleted` (по данным
`/api/auth/progress`). Если условие не сработало — кнопка исчезает. Механика
(`/coach?forceDeep=1` → `sessionMode: 'DEEP'`) рабочая. Нужно вынести вход в
постоянно доступное место.

### 0.4. Логика Supabase (Supabase = Postgres проекта, доступ через Prisma)
Таблицы и что в них пишется:
| Таблица | Что хранит | Когда пишется |
|---|---|---|
| `users.diagnostic_answers` | сырые ответы диагностики | по ходу тестов |
| `coach_sessions` | transcript + extractedData, `status`, режим EXPRESS/DEEP | во время/после коуча |
| `diagnostic_results` | по каждому тесту (RIASEC/BFI/ICAR/PVQ/VIA/…): scores + reliability | при завершении теста |
| `digital_profiles.summary` | **объединённый профиль коуч+тесты — источник матчинга** | при сборке отчёта |
| `reports.html_content` | JSON отчёта (не HTML, а сериализованный объект) | при сборке отчёта |
| `reports.pdf_url` | ссылка на PDF | сейчас не заполняется (PDF = client `window.print()`) |

⚠️ **Безопасность:** на всех таблицах **RLS отключён** (`RLS disabled` /
`UNRESTRICTED` на скринах). Для продакшена с данными несовершеннолетних это
критично: утечка anon-ключа = чтение всей базы.

### 0.5. Отчёт скучный «по построению»
Правая колонка — вертикальный стек карточек без иерархии и нарратива;
формулировки академические («Карьерная адаптивность по Савикасу»,
«Склонность к прокрастинации 10/20»). Для школьника это стена терминов.

## 1. Трек A — Многомерный матчинг (методологическое ядро) 🥇 ✅ ГОТОВО (b1cb1d2)

> **Статус:** реализовано. `matchProfessions(MatchProfile)` считает 5 осей
> (интересы 0.30, личность 0.15, ценности 0.15, сильные стороны 0.15,
> когнитивная 0.10) с ре-нормировкой весов по доступным сигналам; экспресс-путь
> RIASEC+BigFive работает; когнитивная ось — мягкий guard-rail (пол 0.5); каждый
> матч несёт `breakdown[]` (объяснимость). next-question передаёт полный профиль,
> breakdown идёт в отчёт. Оси Гарднера/предметов/мотивации-коуча **отложены** —
> нет чистого источника у ученика (при появлении добавятся тем же механизмом).
> Отображение breakdown на карточке — в Треке B.


**Цель:** матчинг использует все валидно собранные оси профиля, а не 2 из 8.
Плюс объяснимость: карточка показывает, *по каким* характеристикам совпадение.

### A1. Спроектировать веса и формулу
Новый `fit` как взвешенная сумма нормированных (0–1) под-фитов, с **переносом
весов на доступные сигналы** (graceful degradation по покрытию):

| Ось | Источник ученика | Поле паспорта | Базовый вес |
|---|---|---|---|
| Интересы (Holland) | `riasec` | `riasec` | 0.30 |
| Личность | `bigFive` | `bigFive.traits` | 0.15 |
| Ценности | PVQ `topValues` | `values` | 0.15 |
| Сильные стороны | VIA `signatureStrengths` | `viaFit` | 0.15 |
| Когнитивная нагрузка | ICAR `band` | `cognitiveDemand` | 0.10 |
| Интеллекты Гарднера | (из RIASEC/интересов, при наличии) | `gardner` | 0.05 |
| Предметы/склонности | hobbies/subjects | `subjects` | 0.05 |
| Ценности коуча/мечты | коуч `values`,`dreams` (эвристика) | — (бонус) | 0.05 |

Правила:
- Если по оси нет данных ученика ИЛИ нет поля у профессии → ось исключается,
  веса **ре-нормируются** по оставшимся. Экспресс-путь (только RIASEC+BigFive)
  продолжит работать.
- `cognitiveDemand` — **мягкий guard-rail**: сильное несоответствие
  (developing ученик ↔ high-demand профессия) штрафует, но не обнуляет —
  не «отрезаем мечту», а честно показываем разрыв.
- Веса вынести в конфиг-константы с комментариями-обоснованиями.

### A2. Реализовать под-фиты
Добавить чистые функции в `professionMatch.ts`:
`valuesFit`, `viaFit`, `cognitiveFit`, `gardnerFit`, `subjectsFit` — по образцу
существующих `riasecFit`/`bigFiveFit` (нормировка 0–1, нейтраль 0.5 при пропуске).
Расширить сигнатуру `matchProfessions(profile)` — принимать не 2 аргумента, а
объект профиля (riasec, bigFive, topValues, signatureStrengths, icarBand, …).

### A3. Объяснимость (explainability)
`ProfessionMatch` дополнить полем `breakdown: { axis, label, score, weight }[]`.
Пробросить в отчёт, чтобы карточка (Трек B) показывала: «Интересы 92 %,
Ценности 80 %, Сильные стороны 75 %…». Это и доверие, и «не чёрный ящик».

### A4. Обновить вызовы и тесты
- `app/api/v1/diagnostic/next-question/route.ts` — передать в `topArchetypes`
  полный профиль (сейчас передаёт только riasec+bigFive).
- Расширить `professionMatch.test.ts`: монотонность, ре-нормировка при пропусках,
  граничные случаи (0 осей → нейтраль), детерминизм, guard-rail когнитивной оси.

**Файлы:** `professionMatch.ts`, `next-question/route.ts`, `professionMatch.test.ts`,
возможно `layers.ts` (хелпер извлечения осей из `SummaryProfile`).
**Критерий готовности:** матчинг учитывает ≥5 осей; экспресс-путь не сломан;
на карточке видна разбивка; тесты зелёные; tsc чист.
**Риск:** изменение ранжирования у существующих профилей — ок, это и есть цель;
зафиксировать golden-тестом «профиль → топ-5», чтобы регрессии были видимы.

## 2. Трек B — «Колесо» профессий: раскрытие по клику 🥈

**Цель (явный запрос):** на вкладке «Карьерные рекомендации» карточка
**по клику** разворачивается, показывая описание, удивительный факт, «почему
подходит» и близкие специализации.

### B1. Добавить поле `fact` в паспорт
- Расширить `interface Profession`: `fact?: string` — короткий (1–2 предложения)
  удивительный/цепляющий факт о профессии. Правило текста то же, что для
  `why`/`summary`: без «ИИ»/«искусственный интеллект».
- Наполнить факты для 205 профессий (курировано, батчами, как каталог).
  Факты — проверяемые и «вау» для подростка (история, зарплатные вилки, необычные
  задачи, известные представители, неожиданные пересечения).
- Тест-инвариант: у каждой профессии есть непустой `fact`, без запрещённых слов.

### B2. Переделать `RpgProfessionCard` в раскрывающуюся
- Свёрнутое состояние: название, %, тир-бейдж (🟢/🔵/🟣), 1 строка тизера.
- По клику (кнопка/`aria-expanded`, доступно с клавиатуры) — плавное раскрытие:
  - `summary` (что делает),
  - блок «💡 А ты знал?» с `fact`,
  - «Почему тебе подходит» — персональный `why` + разбивка из A3,
  - RIASEC-мини-профиль (уже есть),
  - «Близкие специализации» (веер `variants`, уже есть),
  - школьные предметы (`subjects`) и ключевые навыки (`skills`).
- Анимация раскрытия — уважать `prefers-reduced-motion`.
- «Колесо»: опционально — радиальная раскладка топ-профессий, где клик по
  сектору открывает карточку. Реализовать после базового аккордеона (риск/объём).

**Файлы:** `professions_db.ts` (+`fact`, +наполнение), `RpgProfessionCard.tsx`,
`report/page.tsx` (проброс новых полей), тип `Profession` в отчёте.
**Критерий:** клик разворачивает/сворачивает; факт и описание видны; работает
с клавиатуры; в свёрнутом виде список компактный, не портянка.

## 3. Трек C — Починить PDF и вернуть глубокую сессию 🥉

### C1. Единый источник для экрана и печати
Убрать расхождение двух вёрсток. Два варианта (выбрать при реализации):

- Гарант от повторного расхождения: тест/чек-лист «каждая секция отчёта имеет

### C3. Постоянный вход в глубокую сессию

- Проверить, что `/api/auth/progress` реально отдаёт `coachSessionMode` и

- E2E-проверка сценария: экспресс → тест → «пройти глубокую» → DEEP → отчёт с

**Файлы:** `report/page.tsx`, `assessment/page.tsx`, `api/auth/progress`,

**Критерий:** в PDF есть все секции экрана; глубокая сессия доступна явной

## 4. Трек D — Антипортянка: нарративный отчёт для школьника

**Цель:** превратить стену терминов в понятную «историю о себе».

### D1. Переструктурировать в главы
Вместо плоского стека — 5 глав-историй с крупными акцентами, по одному инсайту
на экран:
1. **«Кто ты»** — код призвания + карта способностей (RIASEC-радар) простым языком.
2. **«Твои суперсилы»** — VIA signatureStrengths + Big Five как сильные стороны.
3. **«Что тебя драйвит»** — ценности (PVQ) + мечты/кумиры из коуча.
4. **«Как ты действуешь»** — когнитивный стиль, Савикас, прокрастинация — как
   практические советы, не диагнозы.
5. **«Куда это ведёт»** — карьерные рекомендации (Трек B) + первый шаг из коуча.

### D2. Язык и подача
- Переписать заголовки/подписи под подростка (сохранив научный термин мелким
  подзаголовком «для родителей/куратора»).
- Прогресс/навигация по главам (якоря, «читать далее»), чтобы не скроллить портянку.
- Визуальные акценты вместо процентов-в-ряд: иконки, короткие «карточки-инсайты».
- Вкладка «Для родителей» (`activeTab==='parent'` уже заложена) — отдельный,
  более академичный срез.

**Файлы:** `report/page.tsx` и его под-компоненты, возможно новые в
`app/components/report/`.
**Критерий:** первый экран — не список из 8 блоков; каждый раздел читается как
законченная мысль; есть навигация; язык проверен «на школьнике».

## 6. Последовательность и обоснование

Делаем по одному треку до конца, с проверкой (tsc + тесты + запуск), затем показ.

1. **A (матчинг)** — ядро ценности; без него «рекомендации» необоснованны, а
   объяснимость из A нужна карточке в B. Делаем первым.
2. **B (колесо)** — прямой запрос владельца; использует разбивку из A.
3. **C (PDF + глубокая сессия)** — чинит осязаемые «потери» (пустой PDF,
   пропавшая кнопка); независим, можно и раньше, если приоритет сместится.
4. **D (нарратив)** — самый крупный по UX; опирается на готовые A/B (есть что
   красиво подавать).
5. **E (RLS)** — параллельно, до запуска; часть — на стороне владельца.
6. **F (данные)** — фоновый долг, по мере надобности.
