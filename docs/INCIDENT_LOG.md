# 📋 Лог инцидентов (Incident Log)

В этом файле фиксируются все решенные проблемы, баги и критические исправления в проекте с указанием симптомов, причин и принятых решений.

## [INC-001] Санитария API-ключей и централизация конфигурации

**Дата:** 16 июля 2026  
**Статус:** Решено  
**Симптомы:**
- В исходном коде (`app/lib/gemini.ts` и `app/lib/env.ts`) находились захардкоженные резервные API-ключи.
- Риск утечки системного ключа в публичный репозиторий GitHub.
- Дублирование логики замены невалидных ключей в двух независимых модулях.

**Причина:**
Временное хардкод-решение для обеспечения непрерывности работы при блокировке основного ключа во время тестирования.

**Решение:**
1. Полностью удален хардкод секретов из кодовой базы.
2. Проверка и санитария ключа перенесена исключительно в `app/lib/env.ts`.
3. В `app/lib/gemini.ts` импортируется валидированное свойство `env.PROXYAPI_KEY`.
4. Во время сборки Next.js на CI/CD (Vercel) отсутствие ключа выводится как `console.warn`, что не ломает билд, но в рантайме бросается критический `Error`.

## [INC-002] Дублирование логики обхода PgBouncer

**Дата:** 16 июля 2026  
**Статус:** Решено  
**Симптомы:**
Одинаковая замена порта БД `:6543/` → `:5432/` и очистка `?pgbouncer=true` происходила и в `env.ts`, и в `prisma.ts`.

**Причина:**
Несогласованные изменения при фиксе проблем с подключением к Supabase.

**Решение:**
1. Логика PgBouncer-замены вынесена полностью в `env.ts`.
2. Из `prisma.ts` удалена избыточная функция `getDbUrl`, клиент БД инициализируется напрямую через `env.DATABASE_URL`.

## [INC-003] Отсутствие глобальных страниц ошибок и загрузки в Next.js App Router

**Дата:** 16 июля 2026  
**Статус:** Решено

## [INC-004] Карточка Колеса/Пирамиды пропадала на промежуточных ширинах окна

**Дата:** 19 июля 2026
**Статус:** Решено
**Симптомы:**
Пользователь прислал скриншот `/coach`: карточка «Колесо талантов» отображалась
не рядом с чатом, а в нечитаемом виде, будто вообще выпадала из макета —
воспроизводится на неразвёрнутом на весь экран окне браузера (см. `docs/18`
P1.4, отмеченный выполненным 18.07.2026, — регрессия обнаружена уже после
этой отметки).

**Причина:**
`app/coach/page.tsx`: правая колонка (`WheelOfVocation`/`PyramidOfAlignment`)
имела класс `hidden lg:flex` — ниже брейкпоинта `lg` (1024px CSS-px) колонка
полностью убиралась из раскладки (`display:none`), без какого-либо
промежуточного состояния. Единственный доступ на узких экранах — плавающая
кнопка, открывающая модалку на весь экран. На ширинах чуть меньше 1024px
(частый случай — свёрнутое, не на весь экран окно браузера) пользователь не
видел карточку вообще, только чат, что выглядело как «пропала»/«переехала».

**Решение:**
1. Правая колонка теперь всегда в потоке документа: на `lg`+ занимает
   `lg:col-span-5` рядом с чатом (без изменений), ниже `lg` — обычная
   инлайн-карточка ограниченной высоты (`h-[70vh]`) сразу под чатом.
2. Убраны плавающая кнопка и мобильная модалка (`showVocationModal`) —
   избыточны при всегда видимой инлайн-карточке; полноэкранный zoom по клику
   остался единственным и универсальным способом развернуть визуализацию.
3. Контейнер колонок (`overflow-y-auto` ниже `lg`) больше не обрезает контент
   при вертикальном стеке на узких экранах.

Коммит: `fix(coach): keep Wheel/Pyramid card always in flow instead of hidden <1024px`.

**Апдейт (19.07.2026, тот же день):** диагноз выше был неполным. После
нескольких раундов правок (свои же более ранние коммиты этой же сессии по
ошибке приняты за «параллельного агента» — см. `feedback_careful-layout-edits`
в памяти) найдена настоящая первопричина через TypeScript compiler API:
в JSX не хватало одного `</div>` после блока ввода сообщения, из-за чего
колонка Колеса была не соседом чата в grid, а буквально вложена внутрь его
карточки — этот баг существовал уже в коммите `380c728`, то есть до всех
сегодняшних правок. Финальный фикс: коммит `b210591`
(`fix(coach): add missing closing div — wheel was structurally nested inside chat card`),
проверено через AST-обход, живой DOM (`grid.children.length === 2`) и
скриншотом бок-о-бок карточек.

## [INC-005] ИИ-коуч зациклился на шаге 2 — сбой у провайдера ProxyAPI/Freemodel.dev

**Дата:** 19 июля 2026
**Статус:** Решено
**Симптомы:**
Пользователь застрял в коуч-сессии на «Шаг 2: Подключение канала связи» —
коуч раз за разом повторял одну и ту же фразу «давай подключим канал связи»,
игнорируя любые ответы («уже подключил», «15» и т.д.). Выглядело как
поломка авторизации.

**Причина:**
Не авторизация — прямая проверка в БД показала, что телефон/telegramId
пользователя сохранены корректно (`hasPhone` было `true`). Реальная причина:
вызов ИИ-провайдера (`api.freemodel.dev`, через `PROXYAPI_KEY`) стабильно
падал — то `500 Failed to start container: Maximum number of running
container instances exceeded`, то `401 Unauthorized` на один и тот же ключ
подряд (нестабильность на стороне провайдера). Любой сбой генерации в
`app/lib/gemini.ts` тихо откатывался на статический `FALLBACK_REPLIES[шаг]`
— для шага 2 это ровно фраза про подключение канала, поэтому и создавалось
впечатление бесконечного цикла именно на этом шаге.

**Решение:**
1. Владелец выдал новый `PROXYAPI_KEY` — старый ключ совпадал с
   захардкоженным в `env.ts` идентификатором известного скомпрометированного
   ключа из INC-001.
2. У нового ключа не оказалось доступа к `claude-sonnet-4-20250514` (только
   к линейке GPT-5.x) — `DEFAULT_MODEL` в `app/lib/gemini.ts` переключён на
   `gpt-5.5` по решению владельца. Коммит `9ee877e`.
3. Владелец выдал ещё 2 резервных ключа. Вместо старой логики (один статичный
   fallback, срабатывающий только для одного конкретного мёртвого ключа)
   реализован honest перебор: `env.PROXYAPI_KEYS` — массив всех непустых
   ключей (`PROXYAPI_KEY`, `_FALLBACK`, `_FALLBACK2`), `callFreemodelWithRetry`
   в `gemini.ts` пробует каждый по очереди при сбое. Коммит `4a6b727`.
4. Проверено: невалидный ключ → 401 → переход на следующий; оба резервных
   ключа отдельно дают 200; полный E2E-прогон через локальный сервер
   (старт сессии → свободный текст → ИИ-экстракция) подтвердил рабочую
   связку ключ+модель по всему стеку, не только в изолированном curl.

**Важно, не забыть:** `.env` не в git — новые ключи (`PROXYAPI_KEY`,
`PROXYAPI_KEY_FALLBACK`, `PROXYAPI_KEY_FALLBACK2`) обновлены только локально.
На Vercel (Production env) их нужно добавить/обновить вручную, иначе прод
продолжит использовать старый нерабочий ключ.
