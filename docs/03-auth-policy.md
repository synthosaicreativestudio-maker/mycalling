# Политика авторизации, безопасности и потока пользователя

В системе «МоёПризвание» реализован строгий раздельный контур для создания новых учетных записей и входа в личный кабинет. Это защищает систему от флуда аккаунтами и гарантирует, что каждый пользователь проходит методологическую коуч-сессию.

> [!IMPORTANT]
> **Терминология**: Слово «регистрация» запрещено во всех пользовательских текстах. Используются мягкие формулировки: «подключение», «привязка профиля», «сохранение результатов».

---

## 1. Потоки создания учётных записей

### Поток А: Через коуч-сессию (основной)
1. Пользователь заходит на главную страницу и начинает диалог с наставником Романом.
2. На **Шаге 0** создаётся гостевая учетная запись `User` (с email `guest_{random}@moiprizvanie.ru`, без пароля).
3. На **Шаге 2** (Подключение канала связи) пользователь подключает мессенджер:
   - Нажимает кнопку Telegram или MAX ID → открывается `/auth/link?code={code}&provider={provider}`.
   - Переходит в бот → делится контактом → бэкенд привязывает `telegramId`/`maxUserId` и `phone` к учётной записи.
   - Или пишет номер телефона прямо в чат (fallback без мессенджера).

### Поток Б: Через ботов напрямую (авто-регистрация «на лету»)
Если пользователь отправляет контакт боту Telegram или МАКС, а в БД нет записи с таким номером телефона:
1. Создается запись `User` (роль `STUDENT`, привязывается `telegramId` или `maxUserId`).
2. Создается связанный `Account` с `providerId: "telegram"` или `providerId: "maxid"`.
3. Генерируется сессионный токен Better Auth (срок: 7 дней).
4. Бот присылает ссылку входа: `https://synthosai.ru/api/auth/telegram/callback?token={token}`.

### Запрет прямой регистрации через ЛК
- Страница `/auth` предназначена **только для входа** существующих пользователей.
- Если незарегистрированный пользователь пытается войти через Telegram-виджет, бэкенд возвращает `error=register_denied`.
- В интерфейсе выводится баннер с предложением начать сессию на главной странице.

## 2. Механизм AuthLink (QR-код + поллинг)

### Создание кода
При инициализации коуч-сессии или при посещении `/auth`, фронтенд вызывает `POST /api/auth/link-code`:
- Создаётся запись `AuthLink` со статусом `PENDING`, уникальным `code` и сроком действия 5 минут.
- Код передаётся в QR и ссылки на ботов.

### QR-коды
QR-коды генерируются **локально** с помощью библиотеки `qrcode.react` (компонент `QRCodeSVG`), без зависимости от внешних API. Это гарантирует мгновенный рендер и устраняет проблему с белым блоком при блокировке внешних сервисов.

```tsx
<QRCodeSVG 
  value={qrLink} 
  size={220} 
  fgColor={isTelegram ? '#349ed9' : '#8b5cf6'} 
  level="H"
  includeMargin={true}
/>
```

### Поллинг подтверждения
Фронтенд (coach page и auth page) каждые 2 секунды опрашивает `GET /api/auth/poll?code={code}`:
- `PENDING` → продолжает поллинг
- `COMPLETED` → прекращает поллинг + выполняет действия (см. ниже)
- `EXPIRED` → перезапрашивает новый код

### Действия при COMPLETED на coach page
1. **Устанавливает cookie авторизации** — silent fetch к `/api/auth/telegram/callback?token={sessionToken}` с `redirect: 'manual'` и `credentials: 'include'`.
2. **Продвигает шаг** — отправляет системное сообщение `'Телефон подтвержден через бот'` в коуч API → перескок на Шаг 3.
3. В результате **пользователь сразу авторизован** на сайте — при переходе в ЛК не будет запрашиваться повторный вход.

### Действия при COMPLETED на auth page
- Редирект на `/api/auth/telegram/callback?token={sessionToken}` → установка cookie → редирект в ЛК (`/report`) или на коуч-сессию (`/coach`).

## 3. Логика возвращающихся пользователей в ботах

### Telegram (`/api/webhooks/telegram`)
При получении команды `/start`:
1. Бот проверяет наличие `telegramId` пользователя в БД (`user.findFirst({ where: { telegramId: String(from.id) } })`).
2. **Если пользователь найден и у него есть `phone`** (возвращающийся):
   - Если в `/start` передан код AuthLink → обновляется статус AuthLink на `COMPLETED`, создаётся сессия.
   - Бот отправляет: «👋 С возвращением, {имя}! Вы уже подключены к платформе.» + инлайн-кнопка «🌐 Перейти на платформу».
   - Клавиатура `request_contact` **не показывается**.
3. **Если пользователь не найден** (новый):
   - Бот приветствует и показывает `ReplyKeyboardMarkup` с кнопкой «📱 Поделиться контактом» (`request_contact: true`).
   - При получении контакта: парсинг номера → создание/обновление User → создание сессии → отправка ссылки входа.

### MAX (`/api/webhooks/maxid`)
Аналогичная логика:
1. Проверка `maxUserId` в БД.
2. Возвращающийся → приветствие + кнопка перехода.
3. Новый → кнопка `request_contact` → парсинг vCard → создание User → ссылка входа.

## 4. Callback и редиректы (`/api/auth/telegram/callback`)

При переходе по ссылке быстрого входа:
1. Валидация токена в БД.
2. Проверка срока действия сессии (`expiresAt`).
3. **Определение прогресса пользователя**:
   - Нет коуч-сессии или она не завершена → `/coach`
   - Коуч-сессия завершена, нет диагностики → `/assessment`
   - Диагностика завершена → `/report`
4. Установка HTTP-only cookie `better-auth.session_token`.

```typescript
let redirectPath = '/coach';
const coachSession = await prisma.coachSession.findUnique({
  where: { userId: session.userId }
});
if (coachSession && coachSession.status === 'COMPLETED') {
  redirectPath = '/assessment';
  const diagnosticResult = await prisma.diagnosticResult.findFirst({
    where: { userId: session.userId }
  });
  if (diagnosticResult) {
    redirectPath = '/report';
  }
}
```

## 5. Автоматическая авторизация на сайте

> [!TIP]
> Ключевой принцип: пользователь, подтвердивший свою личность через бота, **не должен повторно авторизовываться** при переходе в Личный кабинет.

**Механизм (реализован в coach page polling):**
1. Пользователь нажимает Telegram/MAX ID в чате → переходит в бот → делится контактом.
2. Webhook обрабатывает контакт → создаёт сессию → обновляет AuthLink на `COMPLETED`.
3. Coach page через poll обнаруживает `COMPLETED` + `sessionToken`.
4. Выполняет **silent fetch** к callback-эндпоинту:
   ```typescript
   await fetch(`/api/auth/telegram/callback?token=${data.sessionToken}`, {
     credentials: 'include',
     redirect: 'manual'
   });
   ```
5. Ответ содержит `Set-Cookie: better-auth.session_token=...` → cookie устанавливается в браузере.
6. При переходе в ЛК (`/auth`) → `authClient.useSession()` обнаруживает сессию → автоматический редирект в `/report`.

## 6. Ограничение кнопки «Поделиться контактом» в Telegram

> [!WARNING]
> Telegram Bot API **не поддерживает** `request_contact` внутри `InlineKeyboardMarkup`. Эта функция доступна **только** в `ReplyKeyboardMarkup`.

**Решение:**
- Кнопка «📱 Поделиться контактом» реализована через `ReplyKeyboardMarkup` с `resize_keyboard: true`, `one_time_keyboard: true` и `input_field_placeholder`.
- После получения контакта клавиатура убирается через `ReplyKeyboardRemove` в следующем сообщении.
- `ReplyKeyboardRemove` и `InlineKeyboardMarkup` (кнопка «Перейти на платформу») отправляются в **двух отдельных сообщениях**, т.к. Telegram не позволяет совмещать их в одном ответе.
