# Политика авторизации и безопасности ЛК

В системе «МоёПризвание» реализован строгий раздельный контур для создания новых учетных записей (регистрации) и входа в личный кабинет (авторизации). Это защищает систему от флуда аккаунтами и гарантирует, что каждый пользователь проходит методологическую коуч-сессию.

---

## 1. Ключевые правила и логика регистрации

### Регистрация только через диалог
- Новые аккаунты создаются исключительно в процессе общения с наставником Романом на [главной странице](file:///Users/verakoroleva/Desktop/идеи/Мое%20Призвание/app/page.tsx).
- В момент указания номера телефона на Этапе 1 коуч-сессии бэкенд нативно генерирует гостевую учетную запись `User` (роль `STUDENT`), связывая с ней сессию.

### Запрет регистрации через кнопку Личного Кабинета
- Страница `/auth` («Личный кабинет») предназначена **только для авторизации** существующих пользователей.
- Если незарегистрированный пользователь пытается войти через Telegram-виджет или МАКС-бот, бэкенд-фильтр в `/api/auth/telegram` пресекает создание нового аккаунта, перенаправляет обратно на `/auth` с ошибкой `error=register_denied`.
- В интерфейсе выводится предупреждающий баннер с предложением начать сессию на главной странице.

## 2. Поддерживаемые методы входа и авторегистрация «на лету»

### Telegram
Авторизация по ссылке входа, полученной от `@moyoprizvanie_bot` после отправки контакта (команда `/start` и кнопка «Поделиться контактом»), сканирования QR-кода или через встроенный браузерный виджет.

### Мессенджер МАКС (MAX)
Авторизация через официального бота `@maxid_bot` в российском мессенджере МАКС (`max.ru`). Пользователь заходит в чат с ботом по ссылке/QR-коду, отправляет свой контакт с помощью специальной кнопки `request_contact` и получает персональную ссылку для быстрого входа в ЛК.

### Механизм автоматической регистрации «на лету» через ботов
Если при отправке контакта боту Telegram или МАКС номер телефона пользователя отсутствует в базе данных, система **автоматически регистрирует его**:
1. Создается запись в таблице `users` (роль `STUDENT`, привязывается соответствующий `telegramId` или `maxUserId`).
2. Создается связанный объект в таблице `account` с `providerId: "telegram"` или `providerId: "maxid"`.
3. Генерируется сессионный токен Better Auth, создается сессия в БД со сроком действия 7 дней.
4. Бот присылает пользователю ссылку входа:
   `https://synthosai.ru/api/auth/telegram/callback?token={sessionToken}`
   (данный колбэк является единым для обоих ботов, так как логика разбора сессий в БД совпадает).

## 3. Сценарий редиректов в Callback

При переходе по ссылке быстрого входа бэкенд в `/api/auth/telegram/callback/route.ts` выполняет валидацию токена:
1. Проверяет наличие токена в БД.
2. Проверяет срок действия сессии (`expiresAt`).
3. Определяет прогресс пользователя:
   - Если у пользователя есть незаконченная коуч-сессия (`CoachSession.status !== 'COMPLETED'`), он автоматически перенаправляется на `/coach` для продолжения диалога.
   - Если коуч-сессия завершена (`CoachSession.status === 'COMPLETED'`), пользователь перенаправляется на `/assessment` для прохождения тестов.
4. Устанавливает HTTP-only куку `better-auth.session_token` на клиенте.

**Пример логики редиректа в коде:**
```typescript
let redirectPath = '/coach';
const coachSession = await prisma.coachSession.findUnique({
  where: { userId: session.userId }
});
if (coachSession && coachSession.status === 'COMPLETED') {
  redirectPath = '/assessment';
}
return NextResponse.redirect(new URL(redirectPath, request.url));
```
