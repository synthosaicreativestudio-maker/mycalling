## Этап 4. Backend / Auth / State

### Исправлено (Critical/High)

1. **🔴 Захардкоженный токен Telegram-бота** в `app/api/auth/telegram/route.ts` —
   реальный токен лежал в коде как fallback и попал в git-историю GitHub.
   Убран (теперь только `process.env`). **Действие для владельца: отозвать токен
   через BotFather и выпустить новый** — из git-истории он никуда не делся.
2. **🔴 Сломанная сессионная кука** в том же роуте: имя `__secure-…` (неверный регистр,
   Better Auth ищет `__Secure-…`) и неподписанный токен (Better Auth принимает только
   `token.signature`). Такую сессию Better Auth не видел вообще. Создан общий модуль
   `app/lib/auth/session-cookies.ts` (подпись + обе куки secure/fallback), роут
   переведён на него.
3. **🔴 Рассогласование шкалы шагов DEEP-сессии** (из отчёта по скриншотам, подтверждено):
   финальный пересчёт шага в `route.ts` выдавал 10–16, тогда как фронтенд, степпер
   пирамиды и ранний блок того же файла работают в шкале 16–22. Последствия:
   в шапке показывались чужие названия шагов («Мерило успеха» во время шага эмоций),
   степпер пирамиды не рендерился, а **анти-залипание для DEEP-шагов было мёртвым кодом**
   (ждало nextStep 16–21, которые никогда не наступали) — это и есть корень жалобы
   «коуч крутит 2–3 уровень пирамиды». Шкала унифицирована на 16–22, двойные маппинги
   в `prompts.ts` убраны, финальное условие разведено по режимам (EXPRESS=16, DEEP=22).
4. **Порог анти-залипания** снижен с 3 до 2 повторов (рекомендация отчёта) — коуч
   делает одну мягкую повторную попытку и двигается дальше.

## Этап 5. База данных (Supabase/PostgreSQL)

- **pg_stat_statements**: топ-10 по времени — целиком внутренние запросы
  Supabase-дашборда/pgbouncer; запросов приложения в топе нет. Проблем нет.
- **Индексы с нулевым сканированием** есть (users_telegram_id_key и др.), но объём
  таблиц — единицы строк: статистика нерепрезентативна, удалять индексы сейчас
  категорически не следует (они понадобятся при росте).
- **Seq scan** доминируют на крошечных таблицах (auth_links: 3 строки) — это оптимальный
  план для Postgres, не проблема.
- **Промежуточные результаты в БД**: каждый ход чата пишется в `coach_sessions`
  (с optimistic-locking `version`), ответы тестов — в `users.diagnosticAnswers` /
  `diagnostic_results`. RAM-сброс Vercel прогресса не теряет. ✅
- **typeof-гарды для JSON**: `getStrLen()` проверяет `typeof val === 'string'`.
  4 места вида `!!x && x.trim()` теоретически уязвимы к не-строке — Low, в план.
