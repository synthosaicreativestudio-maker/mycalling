## 0. Layout-регрессия: карточка Колеса/Пирамиды теряется на промежуточных ширинах

### Discover (уже сделано статическим анализом кода)

`app/coach/page.tsx`:
- Строка 707: `<div className="w-full max-w-7xl flex-1 min-h-0 flex flex-col lg:grid lg:grid-cols-12 gap-6">` — ниже `lg` (1024px) это `flex-col` (одна колонка), на `lg`+ — grid 12 колонок.
- Строка 710: чат — `col-span-1 lg:col-span-7`.
- Строка 992-993: колонка Колеса/Пирамиды — `className="lg:col-span-5 hidden lg:flex ..."` — **ниже 1024px CSS-px эта колонка `display:none` целиком**, без промежуточного состояния между «спрятана» и «рядом с чатом».
- Мобильный доступ к Колесу реализован ТОЛЬКО через плавающую кнопку (строка 1059) → модалка на весь экран с тёмным фоном (строка 1075, `fixed inset-0 bg-black/80 ... lg:hidden`) — это не инлайн-карточка под чатом, а отдельный оверлей поверх контента.

Вывод: в коде нет ни одного пути, где `WheelOfVocation` рендерится как обычная
инлайн-карточка, стоящая в потоке документа сразу под чатом (не оверлей). Три
единственных места рендера — десктопная колонка (`hidden lg:flex`), модалка по
кнопке и полноэкранная модалка по клику/zoom — все они рассчитаны на бинарное
переключение «спрятано / показано», а не на плавную адаптацию. Промежуточные
ширины окна (частый случай — свёрнутое не на весь экран окно браузера, как на
скриншоте пользователя) —slepaya zona: в зависимости от фактической CSS-ширины
в этот момент колонка может быть либо полностью не в DOM визуально (`hidden`),
либо неожиданно попасть в `lg:` состояние с иным поведением grid/flex, чем
ожидает пользователь. Без живого браузера (в этой сессии недоступен
инструмент для интерактивного рендеринга) точный триггер на конкретном
скриншоте подтвердить нельзя — гальюн с золотым лого справа на скриншоте не
соответствует ни одному файлу в `public/assets/` (проверены `telegram-preview.png`,
`brand-logo.png`, `logo-with-text.png` — все на тёмно-синем фоне, не золотом) —
похоже на посторонний элемент рабочего стола/другого окна вне вкладки браузера,
не будет включаться в фикс.

### Plan (что чиню)

Вместо точечного патча — устранить саму причину класса проблем: бинарный
`hidden` без промежуточного состояния.

1. Ввести собственный react-hook `useIsDesktopColumn()` (или переиспользовать
   существующий паттерн определения темы через `matchMedia`, как в
   `layout.tsx:84-93`) с порогом `min-width: 1024px`, синхронизированным с
   Tailwind `lg`.
2. Заменить чисто-CSS `hidden lg:flex` на управляемое состояние: при **любой**
   ширине < `lg` — Колесо/Пирамида показываются как **обычная инлайн-карточка
   в потоке документа** сразу под карточкой чата (внутри того же
   `flex-col`-контейнера на строке 707), с тем же `glass-card`-стилем, что и
   десктопная версия, а не только через модалку по кнопке. Плавающая кнопка
   и полноэкранная zoom-модалка остаются как способ развернуть на весь экран
   поверх этой инлайн-карточки, а не единственный способ её увидеть.
3. На `lg`+ поведение не меняется — колонка `lg:col-span-5` рядом с чатом,
   как сейчас.
4. Затронуть только `app/coach/page.tsx` — компоненты `WheelOfVocation.tsx` и
   `PyramidOfAlignment.tsx` не меняются (переиспользуются как есть).

### Verify

- Ручная проверка в браузере (dev server, `npm run dev`) на ширинах 320,
  480, 768, 900, 1023, 1024, 1280, 1440 CSS-px в обеих темах — Колесо/Пирамида
  видимы и читаемы на КАЖДОЙ ширине, без скачков/пропаданий.
- `npx tsc --noEmit`, `npm run lint`, `npx vitest run`, `npm run build` — зелёные.
- Playwright a11y на `/coach` (после п.2 ниже расширения покрытия) — новый
  инлайн-блок не создаёт нарушений контраста/разметки.

## 1. P1.2 — Speed Index 8.9s на мобильном лендинге

### Discover

Реальная причина (docs/17 ошибочно писал «framer-motion каскады»):
`app/components/DarkCosmicBackground.tsx`, подключённый глобально в
`app/layout.tsx:133` на КАЖДОЙ странице, рендерит **два** `<video autoPlay
loop preload="auto">` одновременно — `background-video-light.mp4` (26 МБ) и
`background-video.mp4` (34 МБ), видимость переключается CSS-классами
`theme-light-only`/`theme-dark-only` (`display:none` в `globals.css:445-453`).
`display:none` не мешает браузеру скачивать/декодировать видео с
`preload="auto"` — **на каждой загрузке страницы качается и играет ~60 МБ
видео**, из них используется только половина. Это и есть Speed Index 8.9s.

### Plan

- В `DarkCosmicBackground.tsx` рендерить только `<video>` активной темы —
  читать тему тем же способом, что инлайн-скрипт в `layout.tsx:84-93`
  (`localStorage.getItem('theme')` / `document.documentElement.dataset.theme`),
  через `useEffect` + `useState`, а не оба видео сразу.
- Добавить проверку `matchMedia('(max-width: 768px)')` и
  `matchMedia('(prefers-reduced-motion: reduce)')`: при срабатывании — не
  рендерить `<video>` вовсе, использовать статический CSS-градиент (для
  тёмной темы добавить аналог уже существующего градиента светлой темы,
  строки 18-23).
- Framer-motion секции `app/page.tsx` не трогаем — не источник проблемы.

### Verify

- Chrome DevTools → Network throttling (Slow 4G) + мобильный viewport —
  видео НЕ должно скачиваться на мобильном.
- Lighthouse mobile на `localhost:3000` (или staging) до/после — ожидаем
  падение Speed Index с 8.9s до ~2-4s.
- Десктоп — визуальный эффект видео-фона сохраняется в обеих темах.

## 2. P3.1 — Sentry

### Discover

`@sentry/nextjs@^10.61.0` в зависимостях, но не инициализирован — нет
`sentry.client.config.ts`, `instrumentation.ts`, `next.config.mjs` не обёрнут
`withSentryConfig`, нет `app/global-error.tsx`.

### Plan

- `sentry.client.config.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`
  — `Sentry.init({ dsn: process.env.NEXT_PUBLIC_SENTRY_DSN, tracesSampleRate: 0.1 })`.
- `instrumentation.ts` в корне — регистрация server/edge конфигов (паттерн
  Next.js 14 App Router).
- `app/global-error.tsx` — обязательный error boundary для App Router.
- Обернуть `next.config.mjs` в `withSentryConfig`.
- Добавить `NEXT_PUBLIC_SENTRY_DSN` в `.env.example` с комментарием.

**Блокер вне кода:** нужен реальный DSN — завести проект на sentry.io и
добавить `SENTRY_DSN`/`NEXT_PUBLIC_SENTRY_DSN` в Vercel env + локальный
`.env`. Код будет готов и неактивен до этого шага — отмечу явно в коммите.

### Verify

- `npm run build` проходит с обёрткой Sentry (даже без реального DSN —
  graceful no-op).
- Если DSN уже добавлен пользователем на момент верификации — тестовая
  ошибка долетает до проекта Sentry (проверяется вручную, не автоматизировано).

## 3. P3.5 — a11y на авторизованных страницах (`/coach`, `/assessment`, `/report`)

### Discover

`tests/a11y.spec.ts` сейчас проверяет только `/` и `/auth` (без сессии).
`app/lib/auth/session-cookies.ts` уже содержит переиспользуемую логику подписи
cookie (`signCookieValue`, `SECURE_SESSION_COOKIE`/`FALLBACK_SESSION_COOKIE`) —
та же схема, что и в проде.

### Plan

- `tests/fixtures/auth-session.ts`: через Prisma создать/переиспользовать
  тестового `User` + `Session` с данными, достаточными для непустого состояния
  `/coach` (сессия начата) → `/assessment` (коуч пройден) → `/report` (есть
  `DiagnosticResult`/`DigitalProfile`), иначе a11y-скан `/report` бессмыслен
  на пустом состоянии.
- Подписать cookie через `signCookieValue` с `process.env.BETTER_AUTH_SECRET`,
  установить оба cookie (`SECURE_SESSION_COOKIE`, `FALLBACK_SESSION_COOKIE`)
  через `context.addCookies(...)` в Playwright перед переходом на страницу.
- Добавить `/coach`, `/assessment`, `/report` в `tests/a11y.spec.ts` (отдельный
  блок с фикстурой, не в общий цикл `PAGES`, т.к. эти три требуют сессии).
- Требует живую БД при прогоне — `DATABASE_URL`/`DIRECT_URL` и так обязательны
  для `npm run start` (см. `app/lib/env.ts`), новых требований к окружению не
  добавляется.

Ручной чек-лист (клавиатурный проход, Monochrome/Blur/Thumb-Zone) — не
автоматизируется, остаётся отдельным пунктом для пользователя, фиксирую в
`docs/18` после мержа как открытый.

### Verify

- `npm run build && npm run start`, затем `npx playwright test` — все
  проходят, включая новые 3 страницы, без critical/serious нарушений.

## 4. P3.6 — a11y в CI

### Plan

`.github/workflows/a11y.yml`:
- Триггер: `pull_request` → `main`.
- Шаги: `npm ci` → `npx prisma generate` → `npx tsc --noEmit` →
  `npm run lint` → `npx vitest run` → `npm run build` → `npm run start &`
  (дождаться готовности порта 3000) → `npx playwright test`.
- Секреты через `secrets.*`: `DATABASE_URL`, `DIRECT_URL`, `PROXYAPI_KEY`,
  `BETTER_AUTH_SECRET`.

**Блокер вне кода:** нужны GitHub repo secrets с тестовыми (не продовыми!)
значениями и, вероятно, отдельная staging-БД для CI, чтобы тесты не писали в
прод. Явно перечислю нужные secrets в коммите/PR — сам завести их в GitHub не
могу.

### Verify

- Тестовый PR — workflow запускается; если секретов ещё нет, упадёт на шаге,
  который их требует, с понятным сообщением, что укажет пользователю, чего
  не хватает.
