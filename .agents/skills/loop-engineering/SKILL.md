# Loop Engineering — Инжиниринг циклов разработки

Этот скилл регламентирует автономное поведение агента при выполнении задач проектирования, кодирования, отладки и верификации кода. Основной принцип — проектирование автономных циклов обратной связи для минимизации ручного вмешательства человека.

## 1. Архитектура цикла (Reason-Act-Observe)

Каждая задача проходит по замкнутому циклу:
1. **Discover (Анализ):** Сбор логов, исследование кодовой базы, проверка внешних источников и стандартов.
2. **Plan (Планирование):** Создание `implementation_plan.md` и согласование с пользователем.
3. **Execute (Исполнение):** Написание/изменение кода.
4. **Verify (Верификация):** Автоматическая проверка кода.
5. **Iterate (Итерация):** Исправление ошибок до 3-х попыток (Self-Correction).

## 2. Автоматизация и проверки (Checkers)

После внесения любых изменений в код агент обязан запустить локальный валидатор/сборщик:
- **Для Python:** Автоматический запуск статического анализатора `ruff check .` (выполнять без запроса разрешения).
- **Для Next.js/TypeScript:** Запуск сборки `npm run build` или линтера `npm run lint` / `next lint`.
- В случае падения сборки — проанализировать stderr лог, внести правки в код и повторить проверку. Лимит самоисправления: **3 итерации**. Если ошибка не решена — предоставить логи пользователю.

## 3. Стейты и память (Memory & State) в Serverless (Vercel)

- **База данных как единственный источник правды:** В serverless-средах любая in-memory память (Map/Set) сбрасывается. Всегда дублируйте состояние активных сессий (тесты, чаты, прогресс) в БД (PostgreSQL) и восстанавливайте кэш при cache miss.
- **Обход PgBouncer:** При возникновении ошибок пула БД Supabase (порт 6543) в Prisma Client перенаправляйте трафик на прямой порт PostgreSQL (`5432`), заменяя подстроку `:6543/` на `:5432/` и удаляя флаг `?pgbouncer=true` в конструкторе.
- **Минимизация проверок кук:** Для API прохождения тестов с UUID-токенами (`session_id`) избегайте избыточной валидации сессионных кук на бэкенде во избежание ложных ошибок 403 Forbidden.

## 4. Слияние гостевых сессий (Soft Merge)

При авторизации анонимного гостя (например, через Telegram/MAX ID) необходимо обеспечить бесшовный перенос данных:
1. **Перенос истории:** До удаления гостя перенесите сессию коуча (`coachSession`), ответы на тесты (`diagnosticAnswers`), результаты (`DiagnosticResult`), цифровой профиль (`DigitalProfile`) и отчет (`Report`) на ID реального пользователя.
2. **MergedInto:** Не удаляйте анонимного гостя синхронно (открытые вкладки браузера гостя продолжат слать запросы с его ID). Присвойте гостю поле `mergedInto = new_user_id`.
3. **Редирект в API:** В API `/api/auth/progress` и API чата проверяйте поле `mergedInto` и прозрачно перенаправляйте контекст на целевого пользователя, пока сессионная кука не обновится в браузере.

## 5. Защита от зацикливания (Loop Protection)

- Механизм защиты от завитков в ИИ-чате коучинга должен иметь порог **не менее 3 повторений** одного и того же шага.
- Автозаполнение полей заглушками типа "Продолжено наставником" должно быть минимизировано — ИИ должен до последнего пытаться мягко извлечь данные от пользователя с помощью наводящих вопросов.

## 6. Участие человека (Human-in-the-loop) и Git

- **Точка согласования:** Агент находится исключительно в режиме диалога и планирования. Запрещено вносить любые изменения в файлы кода до явной команды `"кодируй"` или `"реализуй"`.
- **Git-дисциплина:** После успешной проверки и прохождения тестов/сборки агент обязан автоматически закоммитить изменения и выполнить push в репозиторий (`git push origin main`), не запрашивая подтверждения.
- **Инцидент-логи:** Любая решенная системная проблема должна быть зафиксирована в [docs/INCIDENT_LOG.md](file:///Users/verakoroleva/Desktop/идеи/Мое Призвание/docs/INCIDENT_LOG.md).
