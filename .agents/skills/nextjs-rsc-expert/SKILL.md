# Next.js RSC Expert — Архитектура App Router и управление сессиями

Этот навык описывает стандарты разделения компонентов на серверные (React Server Components) и клиентские (Client Components), а также правила работы с авторизационными куками Better Auth.

## 1. Разделение Server и Client Components

В Next.js App Router все компоненты по умолчанию являются Server Components. Они рендерятся на сервере, не имеют состояния (state) и не могут использовать хуки React (такие как `useEffect`, `useState`).

- **Серверные компоненты (RSC):**
  - Подходят для загрузки данных напрямую из базы данных или внешних API.
  - Содержат конфиденциальные ключи или логику, которая не должна уходить на клиент.
  - Не могут вызывать хуки авторизации на клиенте (например, `authClient.useSession()`).
  
- **Клиентские компоненты (с директивой `'use-client';`):**
  - Подходят для интерактивных элементов (формы, анимации Framer Motion, обработчики кликов).
  - Могут использовать React state, хуки жизненного цикла и клиентские библиотеки Better Auth.
  
- **Правило стыковки:** Если на странице требуется авторизация Better Auth, выносите блок авторизации (имя пользователя, профиль, кнопка «Выйти») в отдельный клиентский компонент (например, `HeaderAuth.tsx`) и рендерите его внутри серверного layout.tsx.

## 2. Работа с куками Better Auth в Server Components и API

Better Auth использует куки для авторизации. В production-среде на HTTPS по умолчанию требуется кука с префиксом `__Secure-` и флагом `secure: true`.

- **Правило ручной записи кук (в кастомных коллбэках или вебхуках):**
  - Устанавливайте одновременно обе куки: `better-auth.session_token` (secure: false) и `__secure-better-auth.session_token` (secure: true). Это исключает проблемы авторизации как на `localhost`, так и на продакшене за прокси (Vercel, CF).
  
- **Формат подписанных кук Better Auth:**
  - При ручной установке куки сессионный токен должен быть подписан по алгоритму better-call:
    `${token}.${Buffer.from(HMAC-SHA256(token, BETTER_AUTH_SECRET)).toString('base64')}`.
  - Простой raw token в куке не пройдет валидацию `getSession()`.
  - Всегда передавайте секрет в конфиг: `betterAuth({ secret: process.env.BETTER_AUTH_SECRET })`.

## 3. SEO и метаданные

- **Правило:** Каждая страница должна иметь уникальные метаданные (title, description), экспортируемые статически или динамически через `generateMetadata` из Server Components.
- На клиентских компонентах (страницы `'use client'`) для динамического изменения title используйте тег `<title>` или внешние хуки управления заголовком.
